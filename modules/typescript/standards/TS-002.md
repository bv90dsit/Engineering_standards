# TS-002: No use of 'any' type

## Standard

TypeScript code SHOULD NOT use the `any` type. Use `unknown` for truly unknown types, or define proper interfaces and type aliases. Exceptions require explicit justification via an inline disable comment.

## Rationale

- `any` defeats the entire type system — it disables type checking for that value and everything it touches
- `any` propagates silently: a single `any` can infect surrounding code, eroding type safety across a module
- `unknown` is the type-safe alternative when you genuinely do not know the type — it forces runtime checks before use
- Proper interfaces make code self-documenting and enable IDE autocompletion

## What good looks like

```typescript
// Bad - any defeats type checking
function processData(data: any): any {
  return data.items.map((item: any) => item.name);
}

// Good - use proper types
interface ApiResponse {
  items: Array<{ name: string; id: number }>;
}

function processData(data: ApiResponse): string[] {
  return data.items.map((item) => item.name);
}

// Good - use unknown when type is genuinely not known
function parseJsonSafely(input: string): unknown {
  return JSON.parse(input);
}
```

When a rare exception is truly necessary:

```typescript
// eslint-disable-next-line @typescript-eslint/no-explicit-any
const legacyLib = require('untyped-legacy-lib') as any;
```

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Automated (CI/CD)** | Regex scan for `: any` in `.ts`/`.tsx` files (excluding tests and type declarations) | Every PR |
| **Automated (ESLint)** | `@typescript-eslint/no-explicit-any` rule | On save / pre-commit |
| **Peer review** | Reviewer checks that any disable comments have valid justification | PR review |

**Primary enforcement: Automated + Peer review.** ESLint and CI catch the pattern. Reviewers verify that any suppression comments are justified.
