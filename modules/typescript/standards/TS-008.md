# TS-008: No @ts-ignore without justification

## Standard

`@ts-ignore` and `@ts-expect-error` MUST NOT be used without an accompanying comment explaining why the type system cannot be satisfied. Unjustified suppressions defeat type safety.

## Rationale

- `@ts-ignore` silently suppresses all type errors on the following line, including legitimate bugs introduced by refactoring
- Without a justification comment, future developers cannot determine whether the suppression is still necessary or was a lazy workaround
- `@ts-expect-error` is slightly better (it errors if there is nothing to suppress), but still requires explanation of the underlying issue
- Unjustified suppressions accumulate over time, creating pockets of untyped code that defeat the purpose of TypeScript
- A mandatory comment forces the author to articulate the problem, often leading to a proper fix instead
- This aligns with the typescript-eslint `ban-ts-comment` rule and supports TS-001 (strict mode)

## What good looks like

```typescript
// @ts-expect-error — third-party library @types/foo is missing the overload
// added in v2.3.0; tracked in JIRA-1234
const result = thirdPartyLib.process(input, { newOption: true });

// @ts-expect-error — React.forwardRef generic inference fails with our
// polymorphic component pattern; see https://github.com/DefinitelyTyped/issues/1234
const WrappedComponent = React.forwardRef(PolymorphicImpl);
```

```typescript
// Even better: avoid @ts-ignore entirely by using type assertions with explanation
const result = thirdPartyLib.process(input, { newOption: true } as ProcessOptions);
```

## What bad looks like

```typescript
// BAD: bare @ts-ignore with no explanation
// @ts-ignore
const value = someFunction(arg);

// BAD: @ts-expect-error with no comment
// @ts-expect-error
obj.undocumentedMethod();

// BAD: vague non-explanation
// @ts-ignore — doesn't work
const x = broken.thing;

// BAD: @ts-ignore used to avoid fixing actual type errors
// @ts-ignore
const user: User = JSON.parse(response.body);  // Should use validation/parsing
```

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Automated (CI/CD)** | Regex or ESLint rule detects `@ts-ignore` / `@ts-expect-error` without a trailing comment | Every PR |
| **Peer review** | Reviewer verifies the justification comment is meaningful and the suppression is necessary | PR review |

**Primary enforcement: Automated + Peer review.** The typescript-eslint `ban-ts-comment` rule (configured to require description) catches bare suppressions automatically. Reviewers verify that justification comments are meaningful and that a proper fix is not feasible.

## Quick reference

```json
// .eslintrc.json — ban-ts-comment rule configuration
{
  "rules": {
    "@typescript-eslint/ban-ts-comment": ["error", {
      "ts-expect-error": "allow-with-description",
      "ts-ignore": "allow-with-description",
      "ts-nocheck": true,
      "ts-check": false,
      "minimumDescriptionLength": 10
    }]
  }
}
```
