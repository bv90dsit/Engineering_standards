# TS-004: No console.log in production code

## Standard

Production TypeScript/JavaScript code SHOULD NOT use `console.log`, `console.warn`, `console.error`, `console.info`, or `console.debug`. Use a structured logger (pino, winston) instead. Console statements are acceptable in test files, build scripts, and CLI tools.

## Rationale

- `console.log` produces unstructured output that cannot be filtered, correlated, or routed to monitoring systems
- Console output in production can leak sensitive data (request bodies, tokens, PII) to container logs
- Structured loggers produce JSON with timestamps, correlation IDs, and severity levels enabling effective observability
- Console methods cannot be configured per-environment (e.g., suppress DEBUG in production)

## What good looks like

- Use a structured logger (pino, winston) instead of console

```typescript
// Bad - unstructured, no levels, no context
console.log('User logged in', userId);
console.error('Payment failed', error);

// Good - structured logging with pino
import pino from 'pino';

const logger = pino({ level: process.env.LOG_LEVEL || 'info' });

logger.info({ userId }, 'User logged in');
logger.error({ err: error, orderId }, 'Payment processing failed');
```

Logger setup should be centralised:

```typescript
// src/lib/logger.ts
import pino from 'pino';

export const logger = pino({
  level: process.env.LOG_LEVEL || 'info',
  formatters: {
    level: (label) => ({ level: label }),
  },
});
```

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Automated (CI/CD)** | Regex scan for `console.(log|warn|error|info|debug)` in `.ts`/`.tsx` files excluding tests and scripts | Every PR |
| **Automated (ESLint)** | `no-console` rule configured to error | On save / pre-commit |

**Primary enforcement: Automated.** ESLint's `no-console` rule and CI grep catch console usage. Test files and build scripts are excluded via ESLint overrides and path filtering.
