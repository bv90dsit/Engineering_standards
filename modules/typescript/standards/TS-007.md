# TS-007: ESLint with strict ruleset configured

## Standard

TypeScript projects SHOULD have ESLint configured with a strict ruleset (e.g., `@typescript-eslint/strict`). ESLint SHOULD run in CI and block merge on errors.

## Rationale

- A strict ESLint configuration catches common bugs (unused variables, implicit type coercion, unsafe any usage) before they reach production
- Consistent linting across a codebase reduces cognitive load during code review — stylistic and correctness issues are handled automatically
- Running ESLint in CI ensures no code is merged that violates the agreed-upon rules, regardless of local editor configuration
- `@typescript-eslint/strict` includes rules that go beyond basic type checking, catching patterns the TypeScript compiler alone does not flag
- Automated enforcement frees reviewers to focus on architecture, logic, and design rather than style nits
- This supports ENG-003 (automated quality gates in CI)

## What good looks like

```json
// .eslintrc.json (flat config or legacy — shown in legacy format for clarity)
{
  "root": true,
  "parser": "@typescript-eslint/parser",
  "parserOptions": {
    "project": "./tsconfig.json"
  },
  "plugins": ["@typescript-eslint"],
  "extends": [
    "eslint:recommended",
    "plugin:@typescript-eslint/strict-type-checked",
    "plugin:@typescript-eslint/stylistic-type-checked"
  ],
  "rules": {
    "@typescript-eslint/no-unused-vars": "error",
    "@typescript-eslint/no-explicit-any": "error",
    "@typescript-eslint/no-floating-promises": "error"
  }
}
```

```typescript
// eslint.config.mjs (flat config — ESLint 9+)
import eslint from '@eslint/js';
import tseslint from 'typescript-eslint';

export default tseslint.config(
  eslint.configs.recommended,
  ...tseslint.configs.strictTypeChecked,
  ...tseslint.configs.stylisticTypeChecked,
  {
    languageOptions: {
      parserOptions: {
        projectService: true,
        tsconfigRootDir: import.meta.dirname,
      },
    },
  },
);
```

```yaml
# CI workflow — ESLint blocks merge on errors
- name: Lint
  run: npx eslint . --max-warnings 0
```

## What bad looks like

```json
// BAD: No ESLint config in a TypeScript project
// (no .eslintrc.json or eslint.config.mjs present)
```

```json
// BAD: Overly permissive — only extends "recommended" without strict rules
{
  "extends": ["eslint:recommended"]
}
```

```yaml
# BAD: ESLint runs but doesn't block merge
- name: Lint
  run: npx eslint . || true
```

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Automated (CI/CD)** | ESLint runs with `--max-warnings 0`; non-zero exit blocks merge | Every PR |

**Primary enforcement: Automated.** CI runs `eslint . --max-warnings 0` on every pull request. The build fails if any errors or warnings are present. Teams adopting this standard should ratchet — start with warnings and tighten to errors over time.

## Quick reference

```bash
# Install
npm install -D eslint @typescript-eslint/parser @typescript-eslint/eslint-plugin typescript-eslint

# Run
npx eslint . --max-warnings 0

# CI script (package.json)
"scripts": {
  "lint": "eslint . --max-warnings 0"
}
```
