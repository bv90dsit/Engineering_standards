# TS-006: Environment variables validated at startup

## Standard

All required environment variables MUST be validated at application startup, not at first use deep in a request. The application must fail fast with a clear error message listing all missing or invalid variables. Use a validation library (Zod, envalid) to define and check the schema.

## Rationale

- Discovering a missing env var mid-request (e.g., a database URL on first query) causes confusing errors, partial failures, and potential data corruption
- Fail-fast at startup gives operators a clear, immediate signal that configuration is wrong — before any traffic is served
- Listing all missing variables at once (not just the first) reduces deploy-fix-deploy cycles
- A typed config object produced by validation gives downstream code compile-time safety for config access
- This is a core principle of 12-factor apps (Factor III: Config) and relates to ARC-002

## What good looks like

```typescript
// src/config.ts — using envalid
import { cleanEnv, str, port, url, bool } from 'envalid';

export const config = cleanEnv(process.env, {
  DATABASE_URL: url({ desc: 'PostgreSQL connection string' }),
  REDIS_URL: url({ desc: 'Redis connection string' }),
  PORT: port({ default: 3000 }),
  NODE_ENV: str({ choices: ['development', 'production', 'test'] }),
  AUTH_SECRET: str({ desc: 'JWT signing secret' }),
  ENABLE_FEATURE_X: bool({ default: false }),
});
```

```typescript
// Alternative — using Zod
import { z } from 'zod';

const envSchema = z.object({
  DATABASE_URL: z.string().url(),
  REDIS_URL: z.string().url(),
  PORT: z.coerce.number().default(3000),
  NODE_ENV: z.enum(['development', 'production', 'test']),
  AUTH_SECRET: z.string().min(32),
  ENABLE_FEATURE_X: z.coerce.boolean().default(false),
});

export const config = envSchema.parse(process.env);
export type Config = z.infer<typeof envSchema>;
```

The config module is imported at the application entry point (e.g., `src/index.ts`) before any other initialisation:

```typescript
// src/index.ts
import { config } from './config'; // Fails fast if env is invalid
import { createApp } from './app';

const app = createApp(config);
app.listen(config.PORT);
```

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Peer review** | Reviewer verifies env vars are validated at startup, not accessed ad-hoc via `process.env` | PR review |
| **Automated (CI/CD)** | Grep for raw `process.env.` access outside the config module | Every PR |
| **Automated (tests)** | Integration test starts the app with missing vars and asserts clear error output | On merge |

**Primary enforcement: Peer review + Automated.** Reviewers check that new env vars are added to the config schema. CI scans for raw `process.env` access outside the centralised config module.
