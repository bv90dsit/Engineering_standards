# TS-005: React components must handle error boundaries

## Standard

React applications SHOULD implement Error Boundary components wrapping major sections of the UI (routes, feature panels, widgets). Without error boundaries, a single component crash blanks the entire page.

## Rationale

- React's default behaviour on an unhandled error is to unmount the entire component tree, showing a blank white page
- Error boundaries catch JavaScript errors in their child component tree and display a fallback UI instead of crashing
- Users lose all context and unsaved work when the full page blanks — error boundaries contain the blast radius
- Granular error boundaries enable partial degradation: one broken widget does not take down the whole page
- Accessibility guidelines require that errors are communicated clearly to users, not as a blank screen

## What good looks like

```typescript
// ErrorBoundary component (class-based, as required by React)
import React, { Component, ErrorInfo, ReactNode } from 'react';

interface Props {
  children: ReactNode;
  fallback?: ReactNode;
}

interface State {
  hasError: boolean;
}

class ErrorBoundary extends Component<Props, State> {
  state: State = { hasError: false };

  static getDerivedStateFromError(): State {
    return { hasError: true };
  }

  componentDidCatch(error: Error, errorInfo: ErrorInfo): void {
    logger.error({ error, errorInfo }, 'React error boundary triggered');
  }

  render(): ReactNode {
    if (this.state.hasError) {
      return this.props.fallback ?? (
        <div role="alert">
          <h2>Something went wrong</h2>
          <p>This section could not be loaded. Please try refreshing the page.</p>
        </div>
      );
    }
    return this.props.children;
  }
}
```

Usage in application layout:

```typescript
function App() {
  return (
    <ErrorBoundary fallback={<FullPageError />}>
      <Header />
      <ErrorBoundary fallback={<SectionError section="main" />}>
        <MainContent />
      </ErrorBoundary>
      <ErrorBoundary fallback={<SectionError section="sidebar" />}>
        <Sidebar />
      </ErrorBoundary>
    </ErrorBoundary>
  );
}
```

## What bad looks like

```typescript
// BAD: No error boundary — a single crash blanks the entire page
function App() {
  return (
    <Header />
    <MainContent />  {/* If this throws, entire page is blank */}
    <Sidebar />
  );
}
```

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Peer review** | Reviewer verifies error boundaries wrap major UI sections (routes, feature areas) | PR review |
| **Peer review** | Reviewer checks that fallback UIs are accessible and informative | PR review |

**Primary enforcement: Peer review.** Reviewers check that new routes and major feature areas are wrapped in error boundaries with appropriate fallback UIs. Automated detection of missing boundaries is not reliable.

## Quick reference

```typescript
// Wrap major UI sections in error boundaries
<ErrorBoundary fallback={<p>Something went wrong.</p>}>
  <MainContent />
</ErrorBoundary>
```
