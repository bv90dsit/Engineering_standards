# JV-008: Structured logging in JSON format

## Standard

Java services SHOULD configure logging to output structured JSON (not plain text). This enables machine parsing by log aggregators (Splunk, ELK, CloudWatch).

## Rationale

- Plain-text log lines require fragile regex parsing in log aggregators; structured JSON fields are parsed natively
- JSON logs enable filtering, alerting, and dashboarding on specific fields (e.g., `traceId`, `userId`, `duration`)
- Structured logging naturally supports context propagation (MDC values become JSON fields)
- Log aggregators (Splunk, ELK/OpenSearch, CloudWatch Logs Insights, Datadog) perform significantly better on structured data
- Correlation across services is trivial when trace/span IDs are first-class JSON fields
- This aligns with 12-Factor App principle XI (Treat logs as event streams) and supports OPS-006

## What good looks like

```xml
<!-- logback-spring.xml — JSON output for production -->
<configuration>
    <springProfile name="production,staging">
        <appender name="STDOUT" class="ch.qos.logback.core.ConsoleAppender">
            <encoder class="net.logstash.logback.encoder.LogstashEncoder">
                <includeMdcKeyName>traceId</includeMdcKeyName>
                <includeMdcKeyName>spanId</includeMdcKeyName>
                <includeMdcKeyName>userId</includeMdcKeyName>
            </encoder>
        </appender>
    </springProfile>

    <springProfile name="local,dev">
        <appender name="STDOUT" class="ch.qos.logback.core.ConsoleAppender">
            <encoder>
                <pattern>%d{HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n</pattern>
            </encoder>
        </appender>
    </springProfile>

    <root level="INFO">
        <appender-ref ref="STDOUT" />
    </root>
</configuration>
```

```java
// Structured key-value pairs in log messages
log.info("Order placed successfully", kv("orderId", order.getId()),
         kv("userId", user.getId()), kv("total", order.getTotal()));
```

Example JSON output:

```json
{
  "@timestamp": "2026-03-15T10:30:00.123Z",
  "level": "INFO",
  "logger_name": "com.example.OrderService",
  "message": "Order placed successfully",
  "orderId": "ORD-12345",
  "userId": "USR-67890",
  "total": 99.99,
  "traceId": "abc123def456"
}
```

## What bad looks like

```xml
<!-- BAD: Plain text pattern in production -->
<encoder>
    <pattern>%d{yyyy-MM-dd HH:mm:ss} %-5level %logger - %msg%n</pattern>
</encoder>
```

```java
// BAD: String concatenation without structured fields
log.info("Order " + orderId + " placed by user " + userId + " for $" + total);
```

```text
// BAD: Resulting plain text log — requires regex to extract fields
2026-03-15 10:30:00 INFO  com.example.OrderService - Order ORD-12345 placed by user USR-67890 for $99.99
```

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Peer review** | Reviewer verifies logback/log4j2 config uses JSON encoder for non-local profiles | PR review |
| **Automated (CI/CD)** | CI step validates logback-spring.xml production profile uses LogstashEncoder or JsonLayout | On merge |

**Primary enforcement: Peer review + Automated.** Reviewers verify that new services include structured logging configuration. A CI check can validate that the production logging config uses a JSON encoder.

## Quick reference

```xml
<!-- logback-spring.xml — add to your Spring Boot project -->
<springProfile name="production">
    <appender name="STDOUT" class="ch.qos.logback.core.ConsoleAppender">
        <encoder class="net.logstash.logback.encoder.LogstashEncoder" />
    </appender>
</springProfile>
```

```groovy
// build.gradle — add logstash-logback-encoder
implementation 'net.logstash.logback:logstash-logback-encoder:7.4'
```
