# JV-003: Use parameterised queries with JPA/JDBC

## Standard

All SQL queries MUST use parameterised statements. String concatenation to build SQL queries is prohibited. Use `@Query` with named parameters (`:param`), `JdbcTemplate` with positional placeholders (`?`), or the Criteria API.

## Rationale

- String concatenation in SQL enables SQL injection — the most common and most dangerous class of web application vulnerability
- Parameterised queries separate code from data, making injection structurally impossible
- Even "internal" queries can become exploitable when upstream data changes or services are composed
- SQL injection is ranked A03:2021 in the OWASP Top 10

## What good looks like

Using Spring Data JPA:

```java
@Query("SELECT u FROM User u WHERE u.email = :email")
Optional<User> findByEmail(@Param("email") String email);
```

Using JdbcTemplate:

```java
jdbcTemplate.query(
    "SELECT * FROM orders WHERE customer_id = ? AND status = ?",
    new Object[]{customerId, status},
    orderRowMapper
);
```

Using Criteria API:

```java
CriteriaBuilder cb = entityManager.getCriteriaBuilder();
CriteriaQuery<User> query = cb.createQuery(User.class);
Root<User> root = query.from(User.class);
query.where(cb.equal(root.get("email"), email));
```

**Avoid:**

```java
// DANGEROUS: SQL injection vulnerability
String sql = "SELECT * FROM users WHERE email = '" + email + "'";
Statement stmt = connection.createStatement();
ResultSet rs = stmt.executeQuery(sql);
```

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Automated (CI/CD)** | Regex detection of SQL keywords followed by string concatenation (`+`) in `.java` files | Every PR |
| **Peer review** | Reviewers verify that all data access uses parameterised queries | Every PR |

**Primary enforcement: Automated + Peer review.** A CI rule flags patterns like `"SELECT..." +` in production code. Reviewers additionally verify that dynamic queries use the Criteria API or parameterised native queries.
