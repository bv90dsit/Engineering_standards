# JV-002: No System.out.println in production code

## Standard

Production Java code SHOULD NOT use `System.out.println`, `System.out.print`, `System.err.println`, or `System.err.print`. Use SLF4J with Logback (or another SLF4J-compatible backend) instead.

## Rationale

- `System.out.println` writes to stdout with no severity level, no timestamp, no structured format, and no routing capability
- Output cannot be filtered by level or directed to different destinations (file, syslog, monitoring platform)
- In containerised environments, unstructured stdout pollutes logs and makes correlation impossible
- SLF4J + Logback provides levels (TRACE, DEBUG, INFO, WARN, ERROR), MDC for context, and configurable appenders
- Print statements left from debugging are a common source of information leakage

## What good looks like

```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

@Service
public class PaymentService {

    private static final Logger log = LoggerFactory.getLogger(PaymentService.class);

    public void processPayment(Payment payment) {
        log.info("Processing payment id={} amount={}", payment.getId(), payment.getAmount());
        // ...
        log.error("Payment failed for id={}", payment.getId(), exception);
    }
}
```

**Avoid:**

```java
public void processPayment(Payment payment) {
    System.out.println("Processing payment: " + payment.getId());  // no level, no format
}
```

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Automated (CI/CD)** | Regex grep for `System.out` or `System.err` in `.java` files excluding test directories | Every PR |

**Primary enforcement: Automated.** A CI step or pre-commit hook scans for `System.(out|err).(print|println)` in production source paths. Test files, CLI tools, and build plugins are excluded.
