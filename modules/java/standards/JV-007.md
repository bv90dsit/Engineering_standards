# JV-007: Use records for DTOs (Java 17+)

## Standard

Data transfer objects SHOULD use Java records instead of classes with manual getters/setters or Lombok `@Data`. Records are immutable, concise, and generate `equals`/`hashCode`/`toString`.

## Rationale

- Records are immutable by default, eliminating accidental mutation of DTOs passed between layers
- A record declaration is typically 1-3 lines vs. 30+ lines for an equivalent class with getters, setters, equals, hashCode, and toString
- Records guarantee correct `equals` and `hashCode` implementations based on all components — manual implementations are frequently buggy
- Records make the data contract explicit: the component list in the header is the single source of truth for the shape of the data
- Unlike Lombok `@Data`, records require no annotation processor, reducing build complexity and IDE issues
- Records are a standard language feature since Java 16 (JEP 395), making them portable across toolchains

## What good looks like

```java
// Concise, immutable, correct equals/hashCode/toString
public record UserDto(
    Long id,
    String email,
    String displayName,
    Instant createdAt
) {}

// With validation in compact constructor
public record CreateOrderRequest(
    String productId,
    int quantity,
    String shippingAddress
) {
    public CreateOrderRequest {
        if (quantity <= 0) throw new IllegalArgumentException("quantity must be positive");
        Objects.requireNonNull(productId, "productId must not be null");
    }
}

// Nested records for complex DTOs
public record OrderResponse(
    String orderId,
    List<LineItemDto> items,
    MoneyDto total,
    Instant placedAt
) {
    public record LineItemDto(String productId, int quantity, MoneyDto unitPrice) {}
    public record MoneyDto(BigDecimal amount, String currency) {}
}
```

## What bad looks like

```java
// BAD: Verbose class with manual boilerplate
public class UserDto {
    private Long id;
    private String email;
    private String displayName;
    private Instant createdAt;

    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    public String getEmail() { return email; }
    public void setEmail(String email) { this.email = email; }
    public String getDisplayName() { return displayName; }
    public void setDisplayName(String displayName) { this.displayName = displayName; }
    public Instant getCreatedAt() { return createdAt; }
    public void setCreatedAt(Instant createdAt) { this.createdAt = createdAt; }

    @Override
    public boolean equals(Object o) { /* ... */ }
    @Override
    public int hashCode() { /* ... */ }
    @Override
    public String toString() { /* ... */ }
}

// BAD: Lombok @Data — mutable, relies on annotation processor
@Data
public class UserDto {
    private Long id;
    private String email;
    private String displayName;
    private Instant createdAt;
}
```

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Peer review** | Reviewer suggests records for new DTO classes that are pure data carriers | PR review |

**Primary enforcement: Peer review.** Reviewers flag new DTO classes that could be records. Existing classes need not be migrated unless actively modified. Classes requiring mutability, inheritance, or JPA entity semantics are exempt.

## Quick reference

```java
// Simple DTO
public record UserDto(Long id, String email, String name) {}

// With compact constructor validation
public record CreateRequest(String name, int quantity) {
    public CreateRequest {
        Objects.requireNonNull(name);
        if (quantity < 1) throw new IllegalArgumentException("quantity must be >= 1");
    }
}
```
