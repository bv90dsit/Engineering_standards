# JV-001: Use constructor injection, not field injection

## Standard

Spring beans SHOULD use constructor injection rather than `@Autowired` field injection. Dependencies should be declared as `final` fields assigned via a constructor.

## Rationale

- `@Autowired` on fields hides dependencies — the class's requirements are not visible from its public API
- Field injection prevents fields from being declared `final`, losing immutability guarantees
- Testing requires reflection or a Spring context to set injected fields; constructor injection allows plain instantiation in unit tests
- Constructor injection makes it immediately obvious when a class has too many dependencies (constructor parameter count is a natural smell)
- The Spring team officially recommends constructor injection as of Spring Framework 4.3+

## What good looks like

```java
@Service
public class OrderService {

    private final OrderRepository orderRepository;
    private final PaymentGateway paymentGateway;

    public OrderService(OrderRepository orderRepository, PaymentGateway paymentGateway) {
        this.orderRepository = orderRepository;
        this.paymentGateway = paymentGateway;
    }
}
```

With Lombok:

```java
@Service
@RequiredArgsConstructor
public class OrderService {

    private final OrderRepository orderRepository;
    private final PaymentGateway paymentGateway;
}
```

**Avoid:**

```java
@Service
public class OrderService {

    @Autowired
    private OrderRepository orderRepository;  // hidden, mutable, hard to test

    @Autowired
    private PaymentGateway paymentGateway;
}
```

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Peer review** | Reviewers verify that new beans use constructor injection and fields are `final` | Every PR |

**Primary enforcement: Peer review.** Reviewers should flag any new usage of `@Autowired` on fields. Optional: configure ArchUnit or IntelliJ inspections to detect field injection automatically.
