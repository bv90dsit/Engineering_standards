# JV-006: Use SLF4J for logging, not java.util.logging or Log4j directly

## Standard

Java services SHOULD use SLF4J (`org.slf4j.Logger` / `org.slf4j.LoggerFactory`) as the logging API. Direct usage of `java.util.logging` (JUL) or `org.apache.log4j` (Log4j) is prohibited in production code.

## Rationale

- SLF4J is a logging facade that decouples application code from the underlying logging implementation
- Direct Log4j usage risks exposure to version-specific vulnerabilities (notably CVE-2021-44228 / Log4Shell) and makes it harder to swap implementations
- `java.util.logging` has a poor API (no parameterised messages natively), limited configuration options, and inconsistent behaviour across JVM versions
- With SLF4J, the implementation (Logback, Log4j2-via-bridge, etc.) can be changed at deployment time without modifying application code
- Spring Boot uses SLF4J + Logback by default, so direct Log4j usage conflicts with the framework convention

## What good looks like

```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

@Service
public class NotificationService {

    private static final Logger log = LoggerFactory.getLogger(NotificationService.class);

    public void sendNotification(String userId, String message) {
        log.info("Sending notification to user={} type={}", userId, message.getType());
        // ...
        log.debug("Notification sent successfully, id={}", notification.getId());
    }
}
```

With Lombok:

```java
@Slf4j
@Service
public class NotificationService {

    public void sendNotification(String userId, String message) {
        log.info("Sending notification to user={}", userId);
    }
}
```

**Avoid:**

```java
import java.util.logging.Logger;  // JUL — poor API, no facade

import org.apache.log4j.Logger;   // Direct Log4j — version-coupled, vulnerability risk
```

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Automated (CI/CD)** | Regex scan for `import java.util.logging` or `import org.apache.log4j` in `.java` files excluding tests | Every PR |

**Primary enforcement: Automated.** A CI step or pre-commit hook detects imports of `java.util.logging` or `org.apache.log4j` in production source paths. Bridging libraries (e.g., `jul-to-slf4j`) are acceptable in infrastructure configuration.
