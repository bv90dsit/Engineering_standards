# ACC-002: Progressive enhancement

## Standard

Services SHOULD work without JavaScript enabled. Core functionality MUST be available as server-rendered HTML; JavaScript enhances but is not required for basic task completion.

## Rationale

- Users on low-bandwidth connections may not download JS bundles
- Older devices may not execute modern JavaScript
- Corporate proxies may strip or block JavaScript
- Some assistive technologies do not execute JavaScript well
- Improves resilience — service degrades gracefully rather than failing completely

## What good looks like

- Core journey works with JavaScript disabled — e.g. user can complete "Apply for a licence" end-to-end with JS off
- Forms submit via standard HTML form actions with `method="POST"` and server-side validation; JS adds client-side validation as an enhancement
- JavaScript adds polish (autocomplete, character counters, conditional reveals) but the page is usable without it
- No blank white pages when JavaScript fails to load or execute — server renders full HTML on first response
- Server-side rendering for initial page load; client JS hydrates progressively
- GOV.UK Frontend components used with their built-in progressive enhancement (e.g. `govuk-accordion` works as expandable sections with JS, shows all content without)

## What bad looks like

- Blank page with "Please enable JavaScript" message
- Single-page application with no server-side fallback for critical journeys
- Form validation only in JavaScript with no server-side validation
- Navigation that relies entirely on JavaScript event handlers

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Peer review** | Reviewer tests core journey with JavaScript disabled | Every PR affecting user-facing pages |

**Primary enforcement: Peer review.** Reviewers verify that the core user journey functions with JavaScript disabled during code review. Automated testing of no-JS journeys may supplement this but is not the primary mechanism.

## Quick reference

**HTML form that works without JavaScript:**
```html
<form method="POST" action="/apply/name">
  <div class="govuk-form-group">
    <label class="govuk-label" for="full-name">Full name</label>
    <input class="govuk-input" id="full-name" name="fullName" type="text">
  </div>
  <button type="submit" class="govuk-button">Continue</button>
</form>
```
Server-side handler validates and redirects — no JS required for the happy path.

**Cypress test for no-JS journey:**
```bash
# Run Cypress with JS disabled via browser launch flag
npx cypress run --config "chromeWebSecurity=false" --env "blockJs=true"
```

```js
// cypress/support/e2e.js — block JS when env flag set
if (Cypress.env('blockJs')) {
  Cypress.on('window:before:load', (win) => {
    win.document.querySelectorAll('script').forEach(s => s.remove());
  });
}

// cypress/e2e/no-js-journey.cy.js
it('completes core journey without JavaScript', () => {
  cy.visit('/start');
  cy.get('button[type="submit"]').click();
  cy.url().should('include', '/next-step');
});
```

**GOV.UK Frontend Nunjucks progressive enhancement example:**
{% raw %}
```nunjucks
{# Character count — works as plain textarea without JS #}
{{ govukCharacterCount({
  name: "moreDetail",
  id: "more-detail",
  maxlength: 200,
  label: { text: "Provide more detail" }
}) }}
```
{% endraw %}
Without JS the user sees a standard textarea; with JS they get a live character counter.
