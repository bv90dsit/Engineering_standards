# EMG-001: AI-generated code must be reviewed as any other contribution

## Standard

Code produced by AI assistants (Copilot, Claude, Cursor, etc.) MUST go through the same review, testing, and approval process as human-written code. The engineer who submits AI-generated code is accountable for its correctness, security, and maintainability.

## Rationale

- AI tools can introduce subtle bugs, security flaws, and licence-incompatible code
- Accountability must remain with a named human, not delegated to a tool
- Consistent quality bar regardless of how code was produced

## What good looks like

- AI-generated code is committed via normal pull request flow with peer review — no "fast track" for AI output
- Reviewers are told when AI was used — e.g. commit trailers (`Co-authored-by: GitHub Copilot`) or PR template checkbox
- Automated security scanning (SAST, SCA) runs on all code regardless of origin — Snyk/Trivy/Semgrep in CI pipeline
- Engineers can explain and justify every line they submit, including AI-generated lines — "I prompted Copilot for X, verified it does Y, and tested Z"
- AI tools are configured to respect repository `.gitignore` and secrets patterns; tools not on the approved list are not used on OFFICIAL-SENSITIVE+ repos
- Team charter documents which AI tools are approved and under what conditions

## What bad looks like

- Pasting AI output directly into production without review
- Disabling linters or tests because "the AI said it works"
- Using AI tools on OFFICIAL-SENSITIVE or above codebases without approved tooling
- Treating AI suggestions as authoritative for security-sensitive logic

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Peer review** | AI-generated code goes through normal PR flow; reviewer can identify and question AI-generated sections | Every PR |
| **Ways of working** | Team agreement on approved AI tools; logging/attribution of AI usage; no AI on OFFICIAL-SENSITIVE+ without approved tooling | Documented in team charter; reviewed at onboarding |

**Primary enforcement: Peer review + Ways of working.** This standard is fundamentally about human accountability. The pipeline can verify branch protection requires review (a prerequisite), but whether AI code was *actually reviewed critically* is a team-culture question enforced through ways of working.

## Quick reference

**Git commit trailer for AI attribution:**
```
feat: add postcode lookup endpoint

Implemented API integration with OS Places.

Co-authored-by: GitHub Copilot <noreply@github.com>
AI-assisted: yes
```

**Approved AI tools list (placeholder — adapt per org):**
| Tool | Classification | Approved for |
|------|---------------|--------------|
| GitHub Copilot (Business) | OFFICIAL | All repos in the GitHub org |
| Claude Code | OFFICIAL | Repos without OFFICIAL-SENSITIVE data |
| ChatGPT | Not approved | Not connected to internal code |

**Branch protection rule (GitHub) requiring review:**
```bash
# Via GitHub CLI
gh api repos/{owner}/{repo}/rulesets -X POST --input - <<'EOF'
{
  "name": "Require PR review",
  "target": "branch",
  "enforcement": "active",
  "conditions": { "ref_name": { "include": ["~DEFAULT_BRANCH"] } },
  "rules": [
    {
      "type": "pull_request",
      "parameters": {
        "required_approving_review_count": 1,
        "dismiss_stale_reviews_on_push": true,
        "require_last_push_approval": true
      }
    }
  ]
}
EOF
```
