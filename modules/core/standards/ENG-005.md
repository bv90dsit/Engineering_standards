# ENG-005: Technical debt tracked and prioritised

## Standard

Technical debt SHOULD be explicitly tracked (e.g. labelled tickets in the backlog). Teams SHOULD allocate regular capacity (minimum 20% of sprint) to debt reduction. Debt SHOULD be assessed for risk and prioritised accordingly.

## Rationale

- Untracked debt accumulates silently until velocity collapses or security incidents occur
- Government services must be sustainable beyond individual contracts

## What good looks like

- Tech debt backlog with severity ratings (Critical / High / Medium / Low)
- Regular allocation in sprint planning (minimum 20% of capacity)
- Debt items have acceptance criteria (not "tidy up X" — state the measurable outcome)
- Quarterly review of debt trend (count and severity over time)
- New debt acknowledged in PR reviews (label or comment when introducing known shortcuts)
- Debt categorised using the deliberate/inadvertent x reckless/prudent quadrant

## What bad looks like

- "We'll refactor later" (never happens)
- No visibility of debt at programme level
- All capacity goes to features until the service is unmaintainable

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Ways of working** | Debt tracking visible in team backlog; allocation discussed at sprint planning; trend reported at show-and-tell | Ongoing — documented in team charter |

**Primary enforcement: Ways of working.** Debt tracking visible in team backlog; allocation discussed at sprint planning; trend reported at show-and-tell.

## Quick reference

**GitHub label setup for debt tracking:**

```bash
# Create labels for tech debt triage
gh label create "tech-debt" --color "d93f0b" --description "Technical debt item"
gh label create "tech-debt:critical" --color "b60205" --description "Debt causing incidents or blocking delivery"
gh label create "tech-debt:high" --color "d93f0b" --description "Debt significantly impacting velocity"
gh label create "tech-debt:medium" --color "fbca04" --description "Debt causing friction but manageable"
gh label create "tech-debt:low" --color "0e8a16" --description "Debt with minimal current impact"
```

**Debt ticket template (GitHub issue body):**

```markdown
## Technical Debt

**Category:** [Deliberate-Prudent / Deliberate-Reckless / Inadvertent-Prudent / Inadvertent-Reckless]
**Severity:** [Critical / High / Medium / Low]
**Area:** [e.g. API layer, database, CI pipeline]

### What is the debt?
[Describe the shortcut or degradation]

### Impact if not addressed
[What gets worse? Security risk? Velocity? Reliability?]

### Acceptance criteria
- [ ] [Specific measurable outcome, e.g. "Response time < 200ms at p99"]
- [ ] [e.g. "No raw SQL queries remain in controllers"]

### Estimated effort
[T-shirt size: S/M/L/XL]
```

**Quarterly debt review checklist:**
1. Count open debt tickets by severity (compare to previous quarter)
2. Calculate % of sprint capacity spent on debt (target: >= 20%)
3. Identify any Critical/High items older than 2 sprints (escalate)
4. Archive resolved debt items and celebrate progress
