# OPS-003: Incident management process with post-mortems

## Standard

All services MUST have a documented incident management process. All significant incidents (P1/P2) MUST be followed by a blameless post-mortem within 5 working days. Action items from post-mortems MUST be tracked to completion.

## Rationale

- Without structured incident management, response is chaotic and lessons are lost
- Blameless culture encourages honest analysis
- Tracking actions prevents repeat incidents

## What good looks like

- Documented incident process (roles, comms, escalation)
- Severity levels clearly defined:
  - **P1 (Critical):** Complete service outage or data loss affecting all users. Requires immediate response 24/7. Example: production database is unreachable.
  - **P2 (Major):** Degraded service affecting a significant portion of users. Core functionality impaired. Example: payments failing for 30% of users.
  - **P3 (Minor):** Partial degradation with workaround available. Limited user impact. Example: search results slow but functional.
  - **P4 (Low):** Cosmetic issue or minor bug. No significant user impact. Example: incorrect padding on a non-critical page.
- Blameless post-mortem written within 5 working days of P1/P2, following this structure:
  1. **Title** — concise description of the incident
  2. **Date/Duration** — when it started, when it was resolved
  3. **Severity** — P1/P2/P3 with justification
  4. **Impact** — number of users affected, revenue/data loss if applicable
  5. **Timeline** — chronological sequence of events (detection, response, mitigation, resolution)
  6. **Root cause** — the underlying technical or process failure (not "human error")
  7. **Contributing factors** — what allowed the root cause to have this impact
  8. **Action items** — each with owner, priority, and due date
  9. **Lessons learned** — what went well, what could be improved
- Post-mortems published internally (accessible to all engineering staff)
- Action items tracked in the team backlog with due dates
- Trends analysed quarterly (repeat causes, MTTR improvements, severity distribution)
- Incident process follows structured roles: Incident Commander, Communications Lead, Technical Lead (e.g. PagerDuty incident response model)

## What bad looks like

- No defined process — "whoever notices fixes it"
- Blame-focused reviews
- Post-mortems written but actions never done
- Same incident repeats monthly
