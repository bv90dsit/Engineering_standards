# ARC-006: No single points of failure for critical services

## Standard

Critical services SHOULD have no single point of failure. Databases, compute, and external dependencies SHOULD be redundant across availability zones.

## Rationale

- A single point of failure means one component failure takes down the entire service
- Cloud availability zones fail independently — distributing across zones protects against data centre-level outages
- Critical government services often have SLAs of 99.9%+ which require redundancy
- Cost of downtime for public services (citizen impact, reputational damage) typically exceeds cost of redundancy
- GDS expects live services to demonstrate they can survive infrastructure failures

## What good looks like

- Databases run in multi-AZ configuration with automatic failover (e.g. RDS Multi-AZ, Aurora, Cloud SQL HA)
- Compute distributed across at least 2 availability zones with load balancing
- Minimum 2 replicas for all application workloads in production
- External dependency failures handled gracefully (circuit breakers, fallbacks, cached responses)
- Architecture diagrams annotate single points of failure and document mitigations or accepted risks
- Regular failure testing (game days, chaos engineering) validates redundancy works
- Stateless application tier — any instance can handle any request
- Session state stored externally (Redis, database) not in application memory
- Recovery time objective (RTO) and recovery point objective (RPO) documented and tested

## What bad looks like

- Single database instance with no replica — one failure causes total data unavailability
- All compute in a single availability zone
- Application stores session state in memory — instance loss drops active users
- No health checks — traffic continues routing to failed instances
- External API dependency with no timeout, retry, or fallback — upstream outage cascades
- "We'll fix it when it breaks" attitude toward single points of failure in critical services

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Periodic audit** | Architecture reviewed for single points of failure; multi-AZ deployment verified; failure testing evidence reviewed | Service assessment (beta/live) and annual architecture review |

**Primary enforcement: Periodic audit.** Service assessments verify that critical services are designed with redundancy. Architecture reviews identify and track single points of failure with documented mitigations or risk acceptance.

## Quick reference

**AWS Multi-AZ RDS (Terraform):**

```hcl
resource "aws_db_instance" "main" {
  identifier     = "my-service-db"
  engine         = "postgres"
  engine_version = "15.4"
  instance_class = "db.r6g.large"

  # Multi-AZ for automatic failover
  multi_az = true

  # Automated backups (required for Multi-AZ)
  backup_retention_period = 7
  backup_window           = "03:00-04:00"

  # Storage
  allocated_storage     = 100
  max_allocated_storage = 500
  storage_encrypted     = true

  # Networking
  db_subnet_group_name   = aws_db_subnet_group.main.name
  vpc_security_group_ids = [aws_security_group.db.id]

  tags = {
    Service     = "my-service"
    Environment = "production"
  }
}

# Subnets in different AZs
resource "aws_db_subnet_group" "main" {
  name       = "my-service-db-subnets"
  subnet_ids = [aws_subnet.private_az1.id, aws_subnet.private_az2.id]
}
```

**Kubernetes deployment with replicas across zones:**

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-service
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-service
  template:
    metadata:
      labels:
        app: my-service
    spec:
      # Spread pods across availability zones
      topologySpreadConstraints:
        - maxSkew: 1
          topologyKey: topology.kubernetes.io/zone
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels:
              app: my-service
      containers:
        - name: my-service
          image: my-service:latest
          resources:
            requests:
              cpu: "250m"
              memory: "256Mi"
            limits:
              cpu: "500m"
              memory: "512Mi"
          readinessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 10
          livenessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 15
            periodSeconds: 20
---
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: my-service-pdb
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: my-service
```
