# ARC-004: Architecture Decision Records maintained

## Standard

Significant technical decisions SHOULD be documented in Architecture Decision Records (ADRs). ADRs SHOULD be stored in the repository alongside the code they describe.

## Rationale

- Decisions lose context over time — ADRs preserve the "why"
- New team members don't know why choices were made without documentation
- Prevents re-litigating settled decisions
- Provides audit trail for assessors and future maintainers

## What good looks like

- `docs/adr/` directory with numbered markdown files
- Template used consistently (title, status, context, decision, consequences)
- ADR written at decision time not retrospectively
- Superseded ADRs marked but not deleted
- ADRs linked from relevant code or documentation
- Low ceremony — short, focused records not lengthy documents

## What bad looks like

- No record of why technology/approach was chosen
- Decisions only in Slack messages or meeting notes that get lost
- ADRs written months after the decision when context is forgotten
- Overly formal process that discourages recording decisions

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Peer review** | Reviewer asks "does this decision warrant an ADR?" for architectural changes | PR review |

**Primary enforcement: Peer review.** Reviewers prompt authors to write an ADR when a PR introduces a significant architectural or technology choice.
