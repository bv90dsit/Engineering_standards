# ARC-004: Architecture Decision Records maintained

## Standard

Significant technical decisions SHOULD be documented in Architecture Decision Records (ADRs). ADRs SHOULD be stored in the repository alongside the code they describe.

## Rationale

- Decisions lose context over time — ADRs preserve the "why"
- New team members don't know why choices were made without documentation
- Prevents re-litigating settled decisions
- Provides audit trail for assessors and future maintainers

## What good looks like

- `docs/adr/` directory with numbered markdown files
- Template used consistently (title, status, context, decision, consequences)
- ADR written at decision time not retrospectively
- Superseded ADRs marked but not deleted
- ADRs linked from relevant code or documentation
- Low ceremony — short, focused records not lengthy documents
- ADR format follows Nygard's template (title, status, context, decision, consequences)

**Concrete examples:**

A team choosing between PostgreSQL and DynamoDB writes `docs/adr/0003-use-postgresql-for-primary-datastore.md` before starting implementation. The ADR explains the context (relational queries needed, team expertise, cost comparison), the decision (PostgreSQL on RDS), and consequences (need to manage connection pooling, gain transactional integrity). When the team later migrates to Aurora, they write a new ADR and mark 0003 as "Superseded by ADR-0012".

## What bad looks like

- No record of why technology/approach was chosen
- Decisions only in Slack messages or meeting notes that get lost
- ADRs written months after the decision when context is forgotten
- Overly formal process that discourages recording decisions

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Peer review** | Reviewer asks "does this decision warrant an ADR?" for architectural changes | PR review |

**Primary enforcement: Peer review.** Reviewers prompt authors to write an ADR when a PR introduces a significant architectural or technology choice.

## Quick reference

### Full ADR template

```markdown
# [ADR-NNNN] [Short title of decision]

## Status

[Proposed | Accepted | Deprecated | Superseded by ADR-XXXX]

## Context

What is the issue that we're seeing that is motivating this decision or change?

## Decision

What is the change that we're proposing and/or doing?

## Consequences

What becomes easier or more difficult to do because of this change?
```

### adr-tools install and usage

```bash
# Install (macOS)
brew install adr-tools

# Initialise ADR directory in your repo
adr init docs/adr

# Create a new ADR
adr new "Use PostgreSQL for primary datastore"
# -> creates docs/adr/0003-use-postgresql-for-primary-datastore.md

# Mark an ADR as superseded
adr new -s 3 "Migrate to Aurora PostgreSQL"
# -> creates new ADR and updates 0003 status to "Superseded"
```

### Example directory layout

```
my-service/
  docs/
    adr/
      0001-record-architecture-decisions.md
      0002-use-node-js-with-express.md
      0003-use-postgresql-for-primary-datastore.md
      0004-adopt-event-driven-notifications.md
      README.md        # optional index linking to each ADR
  src/
  tests/
```
