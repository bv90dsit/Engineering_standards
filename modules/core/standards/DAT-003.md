# DAT-003: Backups tested with restore drill

## Standard

All production data MUST be backed up. Backup restoration MUST be tested at least quarterly via a documented restore drill. Recovery Point Objective (RPO) and Recovery Time Objective (RTO) MUST be defined and achievable.

## Rationale

- Untested backups are not backups — Schrodinger's backup
- Ransomware and data corruption require proven restore capability
- RPO/RTO must be realistic and validated, not aspirational
- Restore drills surface issues (permissions, tooling, runbook gaps) before a real incident

## What good looks like

- Automated backups with defined RPO (e.g. RPO 1 hour = continuous/WAL archiving; RPO 24h = nightly snapshots)
- Quarterly restore drill to a clean, isolated environment (not the same account as production)
- RTO documented and tested end-to-end, including DNS cutover and application warm-up
- Backup encryption at rest using a KMS key the team controls
- Backup stored in a separate AWS account or region from production
- Restore drill results documented (date, duration, issues found, pass/fail) and shared with the team

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Periodic audit** | Evidence of quarterly restore drill; RPO/RTO defined and tested | Service assessment and live review |
| **Ways of working** | Restore drill in team calendar; results documented | Quarterly |

**Primary enforcement: Periodic audit + Ways of working.** Teams schedule quarterly restore drills and document results. Evidence is reviewed at live service assessment.

## Quick reference

**AWS RDS automated backup (Terraform):**

```hcl
resource "aws_db_instance" "main" {
  identifier              = "myservice-prod"
  engine                  = "postgres"
  backup_retention_period = 14              # days
  backup_window           = "03:00-04:00"   # UTC
  copy_tags_to_snapshot   = true
  storage_encrypted       = true
  kms_key_id              = aws_kms_key.backup.arn

  # Cross-region copy for DR
  replicate_source_db = null
}

resource "aws_db_instance_automated_backups_replication" "dr" {
  source_db_instance_arn = aws_db_instance.main.arn
  kms_key_id             = aws_kms_key.dr_backup.arn
  retention_period       = 14
}
```

**Restore drill checklist:**

- [ ] Identify latest snapshot/backup to restore from
- [ ] Provision isolated environment (separate VPC/account)
- [ ] Restore database from backup
- [ ] Verify row counts and checksums against production
- [ ] Confirm application connects and serves requests
- [ ] Record total elapsed time (compare to RTO)
- [ ] Document any issues or runbook gaps found
- [ ] Clean up isolated environment

**Example pg_restore command:**

```bash
# Restore a custom-format dump to a clean database
pg_restore \
  --host=dr-restore-instance.abcdef.eu-west-2.rds.amazonaws.com \
  --dbname=myservice \
  --username=restore_admin \
  --no-owner \
  --verbose \
  myservice-2026-07-01.dump
```
