# OPS-004: Disaster recovery plan tested annually

## Standard

All production services MUST have a documented disaster recovery (DR) plan. The DR plan MUST be tested at least annually via a full failover exercise. Recovery Time Objective (RTO) and Recovery Point Objective (RPO) MUST be met during the test.

## Rationale

- Untested DR plans are fiction
- Real disasters expose gaps that tabletop exercises miss
- Government services have continuity obligations

## What good looks like

- DR plan documented and versioned in the same repo as the service
- Annual full failover test with success criteria defined in advance
- RTO/RPO defined per service tier, for example:
  - **Tier 1 (Critical):** RTO = 1 hour, RPO = 15 minutes (e.g. payments, auth)
  - **Tier 2 (Important):** RTO = 4 hours, RPO = 1 hour (e.g. case management)
  - **Tier 3 (Standard):** RTO = 24 hours, RPO = 24 hours (e.g. internal reporting)
- Results documented with lessons learned and gaps identified
- DR environment automated (IaC) — infrastructure can be recreated from code
- Team practices DR regularly enough that it's routine, not a fire drill

## What bad looks like

- DR plan is a 3-year-old Word document nobody has read
- "We'll figure it out when it happens"
- DR environment manually configured and drifted
- Last test was before COVID

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Periodic audit** | Evidence of annual DR test with documented results, RTO/RPO met | Service assessment and annual review |

**Primary enforcement: Periodic audit.** Evidence of annual DR test with documented results, RTO/RPO met.

## Quick reference

**RTO/RPO tier definitions:**

| Tier | RTO | RPO | Examples |
|------|-----|-----|----------|
| Tier 1 (Critical) | 1 hour | 15 minutes | Authentication, payments, core API |
| Tier 2 (Important) | 4 hours | 1 hour | Case management, notifications |
| Tier 3 (Standard) | 24 hours | 24 hours | Internal reporting, batch jobs |

**DR test checklist:**

```markdown
- [ ] Pre-test: Confirm DR runbook is up to date
- [ ] Pre-test: Notify stakeholders of test window
- [ ] Execute: Simulate primary region failure (disable primary endpoints)
- [ ] Execute: Verify automatic or manual failover to DR region
- [ ] Execute: Confirm application responds correctly in DR region
- [ ] Execute: Validate data integrity (compare last backup vs expected state)
- [ ] Execute: Measure actual RTO and RPO against targets
- [ ] Post-test: Fail back to primary region
- [ ] Post-test: Document gaps and create action items
- [ ] Post-test: Update DR runbook with lessons learned
```

**AWS cross-region backup (RDS example):**

```bash
# Copy latest automated snapshot to DR region
aws rds copy-db-snapshot \
  --source-db-snapshot-identifier arn:aws:rds:eu-west-1:123456789:snapshot:my-db-daily-backup \
  --target-db-snapshot-identifier my-db-dr-copy-$(date +%Y%m%d) \
  --source-region eu-west-1 \
  --region eu-west-2 \
  --copy-tags
```
