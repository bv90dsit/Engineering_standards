# ARC-005: Resilience patterns (retry, circuit-breaker, timeouts)

## Standard

Services that depend on external APIs or downstream services SHOULD implement resilience patterns: timeouts on all external calls, retries with exponential backoff, circuit breakers to prevent cascade failures, graceful degradation when dependencies are unavailable.

## Rationale

- Government services have diverse dependencies (legacy systems, third-party APIs, shared platforms)
- Without resilience patterns, one downstream failure takes down the whole service
- Users should not see a blank error page because a non-critical dependency is slow
- Cascade failures are expensive and difficult to recover from

## What good looks like

- All HTTP clients have explicit timeouts (connect + read)
- Retries with jitter for transient failures (5xx, timeouts)
- Circuit breaker trips after N failures and recovers gracefully
- Fallback behaviour defined for each dependency
- Chaos testing validates resilience assumptions
- Dashboards show circuit breaker state and retry rates

## What bad looks like

- No timeouts — thread pool exhaustion when downstream is slow
- Infinite retries creating a thundering herd
- No circuit breaker — requests pile up against a dead dependency
- Service returns 500 because a non-critical dependency (analytics, feature flags) is down

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Peer review** | Timeout/retry config on new integrations; fallback behaviour defined | PR review |
| **Periodic audit** | Resilience patterns reviewed; chaos testing results documented | Live service assessment |

**Primary enforcement: Peer review + Periodic audit.** Reviewers check timeout and retry configuration on new integrations. Resilience posture is reviewed at live service assessment, with chaos testing results as evidence.
