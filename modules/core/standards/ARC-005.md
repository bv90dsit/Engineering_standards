# ARC-005: Resilience patterns (retry, circuit-breaker, timeouts)

## Standard

Services that depend on external APIs or downstream services SHOULD implement resilience patterns: timeouts on all external calls, retries with exponential backoff, circuit breakers to prevent cascade failures, graceful degradation when dependencies are unavailable.

## Rationale

- Government services have diverse dependencies (legacy systems, third-party APIs, shared platforms)
- Without resilience patterns, one downstream failure takes down the whole service
- Users should not see a blank error page because a non-critical dependency is slow
- Cascade failures are expensive and difficult to recover from

## What good looks like

- All HTTP clients have explicit timeouts (connect + read)
- Retries with jitter for transient failures (5xx, timeouts)
- Circuit breaker trips after N failures and recovers gracefully
- Fallback behaviour defined for each dependency
- Chaos testing validates resilience assumptions
- Dashboards show circuit breaker state and retry rates
- Release It! (Nygard) for pattern reference

**Concrete examples:**

A payments service calling a downstream fraud-check API configures: 2-second connect timeout, 5-second read timeout, 3 retries with exponential backoff (200ms, 400ms, 800ms + jitter), and a circuit breaker that opens after 5 consecutive failures (half-open after 30s). When the circuit is open, the service returns a cached risk score or queues the check for async processing rather than failing the user's payment.

## What bad looks like

- No timeouts — thread pool exhaustion when downstream is slow
- Infinite retries creating a thundering herd
- No circuit breaker — requests pile up against a dead dependency
- Service returns 500 because a non-critical dependency (analytics, feature flags) is down

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Peer review** | Timeout/retry config on new integrations; fallback behaviour defined | PR review |
| **Periodic audit** | Resilience patterns reviewed; chaos testing results documented | Live service assessment |

**Primary enforcement: Peer review + Periodic audit.** Reviewers check timeout and retry configuration on new integrations. Resilience posture is reviewed at live service assessment, with chaos testing results as evidence.

## Quick reference

### Retry with exponential backoff (Python)

```python
import httpx
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type

@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=0.2, min=0.2, max=2),  # 200ms, 400ms, 800ms
    retry=retry_if_exception_type((httpx.TimeoutException, httpx.HTTPStatusError)),
)
def call_fraud_check(payload: dict) -> dict:
    response = httpx.post(
        "https://fraud.internal/v1/check",
        json=payload,
        timeout=httpx.Timeout(connect=2.0, read=5.0, write=5.0, pool=5.0),
    )
    response.raise_for_status()
    return response.json()
```

### Circuit breaker config (resilience4j - Java/Spring Boot)

```yaml
# application.yml
resilience4j:
  circuitbreaker:
    instances:
      fraudCheck:
        slidingWindowSize: 10
        failureRateThreshold: 50          # open after 50% failures in window
        waitDurationInOpenState: 30s       # half-open after 30s
        permittedNumberOfCallsInHalfOpenState: 3
        slowCallDurationThreshold: 4s
        slowCallRateThreshold: 80
  retry:
    instances:
      fraudCheck:
        maxAttempts: 3
        waitDuration: 200ms
        enableExponentialBackoff: true
        exponentialBackoffMultiplier: 2
        retryExceptions:
          - java.io.IOException
          - java.util.concurrent.TimeoutException
```

### Timeout config for HTTP clients (Node.js with got)

```javascript
const got = require("got");

const client = got.extend({
  prefixUrl: "https://fraud.internal/v1",
  timeout: {
    lookup: 500,     // DNS lookup
    connect: 2000,   // TCP connect
    request: 5000,   // total request (including response body)
  },
  retry: {
    limit: 3,
    methods: ["GET", "POST"],
    statusCodes: [408, 429, 502, 503, 504],
    calculateDelay: ({ attemptCount }) => attemptCount * 200,  // 200ms, 400ms, 600ms
  },
});

// Usage
const result = await client.post("check", { json: payload }).json();
```
