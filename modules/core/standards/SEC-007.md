# SEC-007: Annual penetration testing

## Standard

All production services MUST undergo penetration testing by a CHECK/CREST-certified provider at least annually, and after significant architectural changes. Critical findings MUST be remediated before production deployment.

## Rationale

- Automated tools miss business logic flaws, social engineering vectors, and complex attack chains
- Human testers find what scanners cannot
- Provides independent assurance of security posture
- Required for many government accreditations and service assessments

## What good looks like

- Annual penetration test scheduled and budgeted (typical lead time: 6-8 weeks for procurement + scheduling)
- CHECK or CREST-certified testers engaged (verify certification status via NCSC/CREST directories before contracting)
- Scope document provided to testers covering: external attack surface, authenticated journeys, APIs, infrastructure, and any recent architectural changes
- Findings tracked to remediation with clear ownership and timelines (CRITICAL: 48 hours, HIGH: 7 days, MEDIUM: 30 days)
- Retest confirms fixes are effective (included in original engagement contract, not a separate procurement)
- Results shared with the security team and relevant stakeholders within 5 working days of report receipt
- Significant architectural changes (new auth flows, new public APIs, infrastructure migration) trigger additional testing outside the annual cycle

## What bad looks like

- No penetration testing, or only automated vulnerability scanning presented as a pen test
- Findings ignored or left untracked
- Using uncertified testers for government systems
- Testing only performed once at launch with no follow-up

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Periodic audit** | Evidence of pen test report and remediation tracking reviewed | Service assessment (beta, live) and annual security review |

**Primary enforcement: Periodic audit.** Service assessments verify that penetration testing has been conducted by certified providers and that findings have been tracked to remediation. Annual reviews confirm testing remains current.

## Quick reference

**CHECK/CREST provider lookup:**

| Directory | URL |
|-----------|-----|
| NCSC CHECK companies | https://www.ncsc.gov.uk/information/check-penetration-testing |
| CREST member directory | https://www.crest-approved.org/members/ |

**Scope template (include in pen test brief):**

```
1. Service name and environment URLs (staging + production)
2. Architecture diagram (components, data flows, trust boundaries)
3. Authentication mechanisms and user roles to test
4. APIs (OpenAPI specs or endpoint lists)
5. Infrastructure (cloud provider, IaC references, network topology)
6. Recent changes since last test
7. Out-of-scope items (third-party SaaS, shared infrastructure)
8. Contact details for test coordination and incident escalation
```

**Timeline template:**

| Week | Activity |
|------|----------|
| T-8 | Raise procurement request, confirm budget |
| T-6 | Issue scope document to shortlisted CHECK/CREST providers |
| T-4 | Receive proposals, select provider, sign contract (include retest) |
| T-2 | Pre-test call: confirm access, credentials, environment readiness |
| T+0 | Testing window (typical: 5-10 days depending on scope) |
| T+2 | Draft report received |
| T+3 | Findings triaged, owners assigned, remediation begins |
| T+6 | Retest of CRITICAL/HIGH findings |
