# ARC-002: Twelve-factor application design

## Standard

Services SHOULD follow twelve-factor application principles. At minimum: configuration via environment variables, stateless processes, explicit dependency declaration, and port binding.

## Rationale

- Makes applications portable between environments (dev, staging, prod, cloud, on-prem)
- Enables horizontal scaling without application changes
- Simplifies deployment automation and container orchestration
- Reduces "works in dev, breaks in prod" failures

## What good looks like

- Configuration injected via environment variables or a secrets manager — never hardcoded
- Application is stateless; session state lives in external stores (Redis, database)
- Dependencies explicitly declared (requirements.txt, package.json, go.mod)
- Application binds to a port and serves HTTP — no reliance on app server configuration
- Logs written to stdout/stderr as structured JSON, collected by the platform
- Dev/prod parity maintained (same Docker image, different config)

**Concrete examples:**

A Node.js service reads `DATABASE_URL`, `REDIS_URL`, and `LOG_LEVEL` from the environment. A `.env.example` file is committed (with placeholder values, never real secrets) so developers know which variables to set. The app binds to `PORT` (defaulting to 3000) and emits structured JSON logs to stdout. Session data is stored in Redis, so any instance can handle any request. The same Docker image is promoted from staging to production with only environment variables changing.

## What bad looks like

- Secrets in source code or committed `.env` files
- Application state stored in local filesystem or in-memory between requests
- "It works if you install these 47 things on the server first"
- Logging to local files that no one reads

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Peer review** | No secrets hardcoded; config via env vars; stateless design | Every PR (reviewer checks) |
| **Periodic audit** | Full 12-factor assessment against production architecture | Service assessment |

**Primary enforcement: Peer review + Periodic audit.** Reviewers catch config-in-code and stateful anti-patterns during PR review. Broader architectural compliance (port binding, dev/prod parity, log routing) is assessed holistically during service assessment.

## Quick reference

### Example `.env.example` file

```dotenv
# Application
PORT=3000
NODE_ENV=development
LOG_LEVEL=info

# Database (use connection string from secrets manager in prod)
DATABASE_URL=postgres://localhost:5432/myservice_dev

# Redis (session store)
REDIS_URL=redis://localhost:6379

# External APIs
NOTIFY_API_KEY=test-key-placeholder
NOTIFY_BASE_URL=https://api.notifications.service.gov.uk
```

### Structured logging to stdout (Node.js with pino)

```javascript
const pino = require("pino");
const logger = pino({
  level: process.env.LOG_LEVEL || "info",
  formatters: {
    level(label) { return { level: label }; },
  },
  timestamp: pino.stdTimeFunctions.isoTime,
});

// Usage
logger.info({ requestId: req.id, path: req.url }, "request received");
```

Output:

```json
{"level":"info","time":"2026-03-15T10:22:01.000Z","requestId":"abc-123","path":"/v1/applications","msg":"request received"}
```

### The 12 factors checklist

| # | Factor | Key question |
|---|--------|--------------|
| 1 | Codebase | One repo per service, deployed to multiple environments? |
| 2 | Dependencies | All deps explicitly declared (no implicit system packages)? |
| 3 | Config | All config from env vars / secrets manager (nothing hardcoded)? |
| 4 | Backing services | DBs, queues, caches treated as attached resources via URL? |
| 5 | Build, release, run | Strict separation between build artifact, config, and runtime? |
| 6 | Processes | App is stateless; persistent data in external stores? |
| 7 | Port binding | App self-contained, exports HTTP by binding to a port? |
| 8 | Concurrency | Scale out via process model (multiple instances, not threads)? |
| 9 | Disposability | Fast startup, graceful shutdown on SIGTERM? |
| 10 | Dev/prod parity | Same image/artifact promoted across environments? |
| 11 | Logs | Logs written to stdout as event streams? |
| 12 | Admin processes | One-off tasks run as isolated processes (migrations, scripts)? |
