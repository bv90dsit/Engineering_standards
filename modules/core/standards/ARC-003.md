# ARC-003: APIs follow RESTful conventions and are versioned

## Standard

APIs SHOULD follow RESTful conventions with consistent naming, HTTP verbs, and status codes. APIs MUST be versioned. Public APIs SHOULD have an OpenAPI specification.

## Rationale

- Government services increasingly communicate via APIs — consistency reduces integration cost
- Versioning prevents breaking consumers when APIs evolve
- OpenAPI enables automated tooling, documentation, and client generation
- Consistent conventions reduce onboarding time for developers

## What good looks like

- RESTful URL structure (nouns not verbs, plural resource names)
- Correct HTTP status codes (201 for creation, 404 for not found, etc.)
- API versioned in URL or header (e.g. `/v1/resources`)
- OpenAPI 3.x spec maintained alongside code
- API linting in CI (Spectral or equivalent)
- Rate limiting configured and documented
- Authentication approach documented

**Concrete examples:**

```
GET    /v1/applications          -> 200 (list)
POST   /v1/applications          -> 201 (create, Location header set)
GET    /v1/applications/{id}     -> 200 | 404
PATCH  /v1/applications/{id}     -> 200 | 404 | 422
DELETE /v1/applications/{id}     -> 204 | 404
```

The team maintains `openapi/spec.yaml` in the repo. CI runs Spectral against it on every PR. Breaking changes (removing a field, changing a type) are detected by `oasdiff` and block the build unless a new version is introduced.

## What bad looks like

- Inconsistent naming (mix of camelCase and snake_case)
- Everything returns 200 with error in body
- No versioning — breaking changes pushed without notice
- OpenAPI spec out of date or missing
- No rate limiting on public APIs

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Peer review** | API design reviewed against RESTful conventions; versioning present | PR review for new/changed APIs |
| **Automated (CI/CD)** | Spectral/openapi-lint validates OpenAPI spec; breaking change detection | Every PR |

**Primary enforcement: Peer review + Automated.** API design is reviewed in PRs for adherence to conventions. Automated linting (Spectral) validates the OpenAPI spec in CI and detects breaking changes.

## Quick reference

### Example OpenAPI 3.x spec snippet

```yaml
openapi: "3.0.3"
info:
  title: Applications API
  version: "1.0.0"
paths:
  /v1/applications:
    get:
      summary: List all applications
      operationId: listApplications
      parameters:
        - name: page
          in: query
          schema: { type: integer, default: 1 }
        - name: per_page
          in: query
          schema: { type: integer, default: 25, maximum: 100 }
      responses:
        "200":
          description: A paginated list of applications
          content:
            application/json:
              schema:
                type: object
                properties:
                  data:
                    type: array
                    items:
                      $ref: "#/components/schemas/Application"
                  meta:
                    $ref: "#/components/schemas/Pagination"
        "401":
          description: Unauthorized
```

### Spectral linting in CI

```yaml
# .github/workflows/api-lint.yml
- name: Lint OpenAPI spec
  run: npx @stoplight/spectral-cli lint openapi/spec.yaml --ruleset .spectral.yaml --fail-severity warn
```

Install locally:

```bash
npm install --save-dev @stoplight/spectral-cli
npx spectral lint openapi/spec.yaml
```

### API versioning in URL

```
# Version in URL path (recommended for simplicity)
https://api.service.gov.uk/v1/applications
https://api.service.gov.uk/v2/applications

# When introducing a breaking change, create a new version:
# v1 continues to serve existing consumers (with a sunset date)
# v2 serves consumers who adopt the new contract
```
