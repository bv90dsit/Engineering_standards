# DAT-001: Data classified and labelled

## Standard

All data assets MUST be classified according to the Government Security Classifications policy (OFFICIAL, OFFICIAL-SENSITIVE, SECRET, TOP SECRET). Data stores and APIs MUST be labelled with their classification level.

## Rationale

- Classification determines handling controls — teams cannot apply correct controls if classification is unknown
- Prevents accidental exposure of sensitive data
- Enables proportionate security investment (not every dataset needs the same controls)

## What good looks like

- Data inventory with classification for every data store and API
- APIs annotated with data classification in OpenAPI specs, e.g. `x-data-classification: OFFICIAL-SENSITIVE` on path or schema objects
- Database tables/collections tagged with classification metadata via comments or a registry table
- Classification inherited by derived datasets (e.g. a report drawn from SECRET data is also SECRET)
- Team knows what classification their service handles and can state it without looking it up
- Classification reviewed when data scope changes (new PII field added, third-party feed ingested)

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Ways of working** | Team maintains a data inventory with classification labels | Ongoing |
| **Periodic audit** | All data stores have classification; labels are accurate; handling matches classification | Service assessment and ITHC |

**Primary enforcement: Ways of working + Periodic audit.** Teams maintain their data inventory as part of normal delivery. Classification accuracy is reviewed at service assessment and IT Health Check (ITHC).

## Quick reference

**Government Security Classifications summary:**

| Classification | Handling | Examples |
|----------------|----------|----------|
| OFFICIAL | Standard corporate controls; no special markings required | Most day-to-day government business |
| OFFICIAL-SENSITIVE | Need-to-know; limited sharing; marked clearly | Personal data, commercial-in-confidence |
| SECRET | Approved secure networks only; personnel security clearance SC | Sensitive policy, military capability |
| TOP SECRET | Accredited systems only; DV clearance required | Intelligence, top-level policy |

**OpenAPI spec annotation:**

```yaml
paths:
  /applications/{id}:
    get:
      x-data-classification: OFFICIAL-SENSITIVE
      x-data-classification-reason: "Contains applicant PII"
```

**Database column/table annotation (PostgreSQL):**

```sql
COMMENT ON TABLE applicants IS 'classification=OFFICIAL-SENSITIVE; contains PII';
COMMENT ON COLUMN applicants.national_insurance_number IS 'classification=OFFICIAL-SENSITIVE';
```
