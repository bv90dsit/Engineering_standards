# ARC-001: Cloud-first architecture

## Standard

New services SHOULD be designed for public cloud deployment. Teams SHOULD use managed services over self-hosted infrastructure where they meet security and data classification requirements. Infrastructure SHOULD be defined as code.

## Rationale

- Cloud-first is UK Government policy (since 2013, reaffirmed in Technology Code of Practice)
- Managed services reduce operational burden and improve reliability
- Infrastructure-as-code enables repeatability, auditability, and disaster recovery
- Avoids capital expenditure and long procurement cycles

## What good looks like

- Services deployed to an approved public cloud (AWS, Azure, GCP via Crown Commercial frameworks)
- Infrastructure defined in Terraform, Pulumi, or CloudFormation — never manual console changes
- Managed databases, message queues, and compute (RDS, SQS, Cloud Run, etc.) over self-hosted
- Multi-AZ / multi-region for critical services
- Clear cost allocation and tagging strategy
- Exit strategy documented to avoid vendor lock-in

**Concrete examples:**

A team deploying a new service creates a `terraform/` directory with modules for VPC, ECS, and RDS. All resources are tagged with `team`, `service`, `environment`, and `cost-centre`. The state file is stored in an S3 backend with DynamoDB locking. The service runs on ECS Fargate (managed compute) with RDS PostgreSQL (managed database) across two availability zones. An ADR documents why AWS was chosen and what the exit strategy is (containerised workload, no proprietary APIs in business logic).

## When on-premises is acceptable

- Data classified above OFFICIAL-SENSITIVE with no approved cloud hosting
- Legacy systems mid-migration with a documented cloud migration roadmap
- Specific latency or data residency requirements that cloud cannot meet

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Periodic audit** | Hosting is on approved public cloud; IaC is in version control; exit strategy documented | Service assessment (alpha → beta → live) |

**Primary enforcement: Periodic audit.** Architecture decisions are reviewed at service assessment. The compliance checker can detect IaC files (Terraform, CloudFormation) as a heuristic, but presence of files doesn't confirm cloud-first — this requires human judgement.

## Quick reference

### Terraform backend config (AWS S3 + DynamoDB locking)

```hcl
terraform {
  backend "s3" {
    bucket         = "myorg-terraform-state"
    key            = "services/my-service/terraform.tfstate"
    region         = "eu-west-2"
    encrypt        = true
    dynamodb_table = "terraform-lock"
  }
}
```

### Mandatory tagging strategy

```hcl
locals {
  common_tags = {
    team         = "platform-engineering"
    service      = "payments-api"
    environment  = var.environment        # dev | staging | prod
    cost-centre  = "CC-4421"
    data-class   = "official"
    managed-by   = "terraform"
  }
}
```

### Crown Commercial Service frameworks

Procure cloud hosting through an approved CCS framework:

| Framework | Use for |
|-----------|---------|
| G-Cloud 14 | AWS, Azure, GCP IaaS/PaaS/SaaS (call-off, no further competition) |
| Digital Outcomes and Specialists 6 | Cloud migration consultancy |

Browse frameworks: <https://www.crowncommercial.gov.uk/agreements>
