# ENG-006: Pair/mob programming for complex work

## Standard

Teams COULD use pair programming or mob/ensemble programming for complex, high-risk, or knowledge-critical work. This is a recommendation, not a requirement — teams should adopt the practice where it adds value.

## Rationale

- Reduces knowledge silos (bus factor)
- Catches defects in real-time
- Accelerates onboarding
- Spreads context across the team

## What good looks like

- Pairing normalised for complex stories (e.g. security-critical code, cross-service integration, unfamiliar domains)
- Mob sessions (3-5 people, one driver, time-boxed to 90 mins) for architectural decisions or unfamiliar tech
- No stigma around asking to pair — agreed in team charter
- Team agrees when to pair vs solo (e.g. "pair for anything touching auth or data migration")

## Quick reference

**When to pair (suggested triggers):**

| Situation | Suggested format |
|---|---|
| New team member's first week | Pair on all tasks |
| Security-critical code (auth, crypto, data access) | Pair |
| Production incident investigation | Mob (whole team) |
| Architectural spike or new technology | Mob (time-boxed 90 mins) |
| Routine bug fix in well-understood area | Solo is fine |
| Code review reveals fundamental misunderstanding | Switch to a pairing session |

**Mob programming format:**

- **Driver**: types code, follows instructions from navigators
- **Navigator(s)**: direct the driver, discuss approach
- **Rotate driver** every 10-15 minutes (use a timer)
- **Time-box**: 60-90 minutes max, then break
- **Tools**: VS Code Live Share, Tuple, or screen-share with remote control

**Team charter clause (copy-paste):**

```markdown
## Pairing norms

We pair by default for:
- [ ] Security-sensitive changes
- [ ] Cross-service integration work
- [ ] First contributions from new joiners
- [ ] Incidents and complex debugging

We mob for:
- [ ] Architectural decisions (ADRs)
- [ ] Unfamiliar technology spikes

Anyone can request a pair at any time — no justification needed.
```
