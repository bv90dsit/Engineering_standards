# SEC-008: API rate limiting

## Standard

All public-facing APIs MUST implement rate limiting with documented thresholds. Rate limit responses MUST return 429 with Retry-After header.

## Rationale

- Prevents denial-of-service attacks from overwhelming backend services
- Protects shared infrastructure from noisy-neighbour abuse
- Provides predictable performance for all consumers
- Enables fair usage across API clients
- Rate limit headers allow well-behaved clients to self-throttle

## What good looks like

- Rate limits defined per endpoint or endpoint group based on expected usage patterns (e.g. 100 req/min for read endpoints, 20 req/min for write endpoints)
- Limits documented in API specification (OpenAPI `x-ratelimit-*` extensions or dedicated docs page)
- 429 responses include `Retry-After` header (seconds until client can retry) and a JSON body explaining the limit hit
- Rate limit headers returned on every response: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`
- Different tiers for authenticated vs unauthenticated traffic
- Distributed rate limiting (e.g. Redis-backed) so limits apply across all instances, not per-pod
- Monitoring and alerting on rate limit hits to detect abuse patterns early
- Graceful degradation — rate-limited clients receive a clear error, not a timeout or connection reset

## What bad looks like

- No rate limiting on public APIs, allowing a single client to exhaust resources
- Rate limiting applied but no `Retry-After` header — clients cannot determine when to retry
- Hard-coded limits with no documentation — consumers discover limits only when they hit them
- Per-instance rate limiting that allows total traffic to exceed safe thresholds
- Overly aggressive limits that block legitimate traffic without a mechanism to request higher quotas

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Automated** | CI check verifies rate-limit middleware is configured for all public routes; integration tests confirm 429 + Retry-After returned when limit exceeded | Every PR and pipeline run |
| **Periodic audit** | Rate limit thresholds reviewed against actual traffic patterns; documentation checked for accuracy | Quarterly security review |

**Primary enforcement: Automated + Periodic audit.** CI validates rate-limit configuration is present; quarterly reviews confirm thresholds are appropriate for current traffic volumes.

## Quick reference

**Nginx rate limiting:**

```nginx
# Define rate limit zone (10 requests per second per IP)
limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;

server {
    location /api/ {
        limit_req zone=api_limit burst=20 nodelay;
        limit_req_status 429;

        # Add Retry-After header on 429
        error_page 429 = @rate_limited;
    }

    location @rate_limited {
        default_type application/json;
        add_header Retry-After 60 always;
        return 429 '{"error": "rate_limit_exceeded", "retry_after": 60}';
    }
}
```

**Express rate-limit middleware (express-rate-limit):**

```javascript
const rateLimit = require('express-rate-limit');

const apiLimiter = rateLimit({
  windowMs: 60 * 1000, // 1 minute
  max: 100,            // 100 requests per window
  standardHeaders: true, // Return X-RateLimit-* headers
  legacyHeaders: false,
  handler: (req, res) => {
    res.set('Retry-After', Math.ceil(req.rateLimit.resetTime / 1000));
    res.status(429).json({
      error: 'rate_limit_exceeded',
      message: 'Too many requests, please try again later',
      retry_after: Math.ceil(req.rateLimit.resetTime / 1000),
    });
  },
});

app.use('/api/', apiLimiter);
```

**Spring Boot with Bucket4j:**

```java
@Configuration
public class RateLimitConfig {

    @Bean
    public FilterRegistrationBean<RateLimitFilter> rateLimitFilter() {
        FilterRegistrationBean<RateLimitFilter> registrationBean = new FilterRegistrationBean<>();
        registrationBean.setFilter(new RateLimitFilter());
        registrationBean.addUrlPatterns("/api/*");
        return registrationBean;
    }
}

// Filter using Bucket4j
public class RateLimitFilter extends OncePerRequestFilter {

    private final Map<String, Bucket> buckets = new ConcurrentHashMap<>();

    private Bucket createBucket() {
        return Bucket.builder()
            .addLimit(Bandwidth.classic(100, Refill.intervally(100, Duration.ofMinutes(1))))
            .build();
    }

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                     HttpServletResponse response,
                                     FilterChain chain) throws ServletException, IOException {
        String clientId = request.getRemoteAddr();
        Bucket bucket = buckets.computeIfAbsent(clientId, k -> createBucket());

        ConsumptionProbe probe = bucket.tryConsumeAndReturnRemaining(1);
        if (probe.isConsumed()) {
            response.setHeader("X-RateLimit-Remaining", String.valueOf(probe.getRemainingTokens()));
            chain.doFilter(request, response);
        } else {
            long waitSeconds = probe.getNanosToWaitForRefill() / 1_000_000_000;
            response.setHeader("Retry-After", String.valueOf(waitSeconds));
            response.setStatus(429);
            response.getWriter().write("{\"error\":\"rate_limit_exceeded\",\"retry_after\":" + waitSeconds + "}");
        }
    }
}
```
