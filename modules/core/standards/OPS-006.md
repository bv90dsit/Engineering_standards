# OPS-006: Structured logging with correlation IDs

## Standard

All services MUST use structured logging (JSON format). Every request MUST carry a correlation ID propagated across service calls. Logs MUST include: timestamp, level, correlation_id, service_name, message.

## Rationale

- Structured logs are machine-parseable, enabling effective search, aggregation, and alerting in log management systems
- Correlation IDs allow tracing a single user request across multiple microservices
- Without correlation IDs, debugging distributed failures requires manual timestamp correlation — slow and error-prone
- Consistent log fields enable cross-service dashboards and automated anomaly detection
- JSON format integrates natively with CloudWatch, ELK, Splunk, and other log aggregation tools

## What good looks like

- All log output in JSON format with consistent field names across services
- Correlation ID (`X-Correlation-ID` or `X-Request-ID`) generated at the edge (API gateway or first service) and propagated through all downstream calls via HTTP headers
- If no correlation ID is present in an incoming request, the receiving service generates one
- Every log line includes at minimum: `timestamp`, `level`, `correlation_id`, `service_name`, `message`
- Additional context fields (user_id, endpoint, duration_ms) added where appropriate
- Sensitive data (passwords, tokens, PII) never logged — redaction applied at the logging layer
- Log levels used consistently: ERROR for failures requiring action, WARN for degraded conditions, INFO for normal operations, DEBUG for development only

## What bad looks like

- Unstructured text logs (e.g. `console.log("Something happened")`)
- No correlation ID — impossible to trace a request across services
- Inconsistent field names across services (one uses `requestId`, another uses `correlationId`, another uses `traceId`)
- Logging sensitive data (auth tokens, personal data, passwords)
- Excessive DEBUG logging enabled in production, creating noise and storage costs
- Log entries missing timestamps or service identifiers

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Peer review** | Log statements use structured logger; correlation ID propagated in service calls | Every PR |
| **Automated** | CI lint rule checks for `console.log` / `System.out.println` usage; integration tests verify correlation ID propagation | Every pipeline run |

**Primary enforcement: Peer review + Automated.** Code review ensures structured logging patterns are followed; CI linting catches unstructured log statements; integration tests verify correlation ID propagation across service boundaries.

## Quick reference

**Python structlog configuration:**

```python
import structlog
import uuid

structlog.configure(
    processors=[
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.add_log_level,
        structlog.processors.JSONRenderer(),
    ],
    context_class=dict,
    logger_factory=structlog.PrintLoggerFactory(),
)

logger = structlog.get_logger()

# Middleware to extract/generate correlation ID
def correlation_id_middleware(get_response):
    def middleware(request):
        correlation_id = request.headers.get('X-Correlation-ID', str(uuid.uuid4()))
        structlog.contextvars.bind_contextvars(
            correlation_id=correlation_id,
            service_name="my-service",
        )
        response = get_response(request)
        response['X-Correlation-ID'] = correlation_id
        return response
    return middleware

# Usage
logger.info("request_received", endpoint="/api/users", method="GET")
# Output: {"timestamp": "2026-07-05T10:30:00Z", "level": "info", "correlation_id": "abc-123", "service_name": "my-service", "event": "request_received", "endpoint": "/api/users", "method": "GET"}
```

**Node.js pino configuration:**

```javascript
const pino = require('pino');
const { v4: uuidv4 } = require('uuid');

const logger = pino({
  level: process.env.LOG_LEVEL || 'info',
  formatters: {
    level(label) { return { level: label }; },
  },
  base: { service_name: process.env.SERVICE_NAME || 'my-service' },
  timestamp: pino.stdTimeFunctions.isoTime,
});

// Express middleware
function correlationIdMiddleware(req, res, next) {
  const correlationId = req.headers['x-correlation-id'] || uuidv4();
  req.correlationId = correlationId;
  req.log = logger.child({ correlation_id: correlationId });
  res.setHeader('X-Correlation-ID', correlationId);
  next();
}

app.use(correlationIdMiddleware);

// Usage in route handlers
app.get('/api/users', (req, res) => {
  req.log.info({ endpoint: '/api/users', method: 'GET' }, 'request_received');
  // ...
});
```

**Spring Boot logback JSON configuration (logback-spring.xml):**

```xml
<configuration>
  <appender name="JSON" class="ch.qos.logback.core.ConsoleAppender">
    <encoder class="net.logstash.logback.encoder.LogstashEncoder">
      <customFields>{"service_name":"${SERVICE_NAME:-my-service}"}</customFields>
      <includeMdcKeyName>correlation_id</includeMdcKeyName>
    </encoder>
  </appender>

  <root level="INFO">
    <appender-ref ref="JSON" />
  </root>
</configuration>
```

```java
// Filter to set correlation ID in MDC
@Component
public class CorrelationIdFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                     HttpServletResponse response,
                                     FilterChain chain) throws ServletException, IOException {
        String correlationId = request.getHeader("X-Correlation-ID");
        if (correlationId == null || correlationId.isEmpty()) {
            correlationId = UUID.randomUUID().toString();
        }
        MDC.put("correlation_id", correlationId);
        response.setHeader("X-Correlation-ID", correlationId);
        try {
            chain.doFilter(request, response);
        } finally {
            MDC.remove("correlation_id");
        }
    }
}
```
