# OPS-002: Deployment frequency target (weekly minimum)

## Standard

Teams SHOULD deploy to production at least weekly. High-performing teams SHOULD target multiple deploys per day. Deployments SHOULD be automated, repeatable, and low-risk.

## Rationale

- Deployment frequency is one of the four DORA metrics that correlate with organisational performance
- Smaller, more frequent deployments carry less risk and are easier to roll back
- Infrequent deployment creates "big bang" releases that are high-risk and stressful
- Weekly minimum ensures the deployment pipeline stays exercised and working

## What good looks like

- Automated deployment pipeline (commit → build → test → deploy) with no manual gates
- Trunk-based development or short-lived feature branches (< 1 day)
- Feature flags to decouple deployment from release (e.g. `if ld_client.variation("new-checkout", user, False): ...`)
- Zero-downtime deployments (blue/green, canary, rolling) verified by health checks
- Deployment takes minutes, not hours — ideally under 15 minutes commit-to-production
- Any team member can deploy (not gated on a single person)
- Rollback is a single action (e.g. `kubectl rollout undo deployment/my-service`), tested regularly

## What bad looks like

- Monthly or quarterly "release trains"
- Manual deployment runbooks with 50+ steps
- Deployments only happen on Fridays at 5pm (or never on Fridays — both are symptoms)
- Only one person knows how to deploy
- Rollback means "restore from last night's backup"

## DORA benchmarks (2023 State of DevOps)

| Performance level | Deployment frequency |
|-------------------|---------------------|
| Elite | On-demand (multiple per day) |
| High | Weekly to monthly |
| Medium | Monthly to every 6 months |
| Low | Less than once per 6 months |

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Periodic audit** | Deployment frequency measured against DORA benchmarks; deployment pipeline is automated | Quarterly review using deployment logs/metrics |
| **Ways of working** | Team commits to trunk-based development; deploys are not gated on a single person | Documented in team charter; reviewed at retros |

**Primary enforcement: Periodic audit.** Deployment frequency is a runtime metric — it cannot be checked from a repository alone. Teams report it via DORA dashboards (Sleuth, LinearB, Jellyfish, or manual tracking).

## Quick reference

**GitHub Actions deployment pipeline:**

```yaml
name: Deploy
on:
  push:
    branches: [main]

jobs:
  build-test-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build
        run: make build
      - name: Test
        run: make test
      - name: Deploy to staging
        run: make deploy ENV=staging
      - name: Smoke test staging
        run: make smoke-test ENV=staging
      - name: Deploy to production
        run: make deploy ENV=production
```

**Feature flag (LaunchDarkly):**

```python
import ldclient

ld_client = ldclient.get()
user = {"key": user_id, "email": user_email}

if ld_client.variation("new-checkout-flow", user, False):
    return new_checkout()
else:
    return legacy_checkout()
```

**Blue/green deployment (AWS ALB target group swap):**

```yaml
# appspec.yml (CodeDeploy blue/green)
version: 0.0
Resources:
  - TargetService:
      Type: AWS::ECS::Service
      Properties:
        TaskDefinition: <TASK_DEFINITION>
        LoadBalancerInfo:
          ContainerName: "app"
          ContainerPort: 8080
Hooks:
  - AfterAllowTestTraffic: "scripts/run_smoke_tests.sh"
```
