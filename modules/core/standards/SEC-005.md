# SEC-005: Authentication via standards (OAuth2/OIDC)

## Standard

Services requiring authentication MUST use standards-based protocols (OAuth 2.0, OpenID Connect). Custom authentication implementations MUST NOT be built. Government services SHOULD integrate with GOV.UK One Login where applicable.

## Rationale

- Custom auth is the most common source of critical vulnerabilities
- Standards have been reviewed by cryptographers and battle-tested at scale
- Centralised identity reduces credential sprawl and improves user experience
- GOV.UK One Login provides a consistent, accessible, secure sign-in for citizens

## What good looks like

- OAuth 2.0 / OpenID Connect with a trusted Identity Provider (e.g., GOV.UK One Login, Azure AD, Keycloak)
- GOV.UK One Login for citizen-facing government services, using the documented OIDC Authorization Code flow
- MFA enforced for staff-facing services (TOTP or WebAuthn at minimum)
- Session management follows OWASP guidelines:
  - Cookies with `Secure`, `HttpOnly`, `SameSite=Strict` flags
  - Idle timeout of 15 minutes, absolute timeout of 8 hours for staff services
  - Session ID regenerated on privilege elevation
- No passwords stored in custom databases
- Token validation and refresh handled by well-maintained libraries (e.g., `passport` + `openid-client` for Node.js, `spring-security-oauth2-client` for Java)
- ID tokens validated for `iss`, `aud`, `exp`, and `nonce` claims before trusting identity

## What bad looks like

- Rolling your own authentication or session management
- Storing passwords with weak or custom hashing
- Long-lived tokens without refresh or revocation mechanisms
- Authentication logic scattered across the codebase rather than centralised

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Periodic audit** | Architecture review verifies auth approach; penetration test validates implementation | Service assessment (alpha, beta, live) and annual pen test |

**Primary enforcement: Periodic audit.** Authentication architecture is reviewed at service assessments to confirm standards-based protocols are used. Penetration testing validates the implementation is correct and secure.

## Quick reference

**OIDC configuration (Node.js with passport + openid-client):**

```js
const { Issuer } = require('openid-client');
const issuer = await Issuer.discover('https://oidc.integration.account.gov.uk/.well-known/openid-configuration');
const client = new issuer.Client({
  client_id: process.env.OIDC_CLIENT_ID,
  token_endpoint_auth_method: 'private_key_jwt',
  id_token_signed_response_alg: 'ES256',
});
```

**OIDC configuration (Spring Security):**

```yaml
spring:
  security:
    oauth2:
      client:
        registration:
          govuk:
            provider: govuk-one-login
            client-id: ${OIDC_CLIENT_ID}
            authorization-grant-type: authorization_code
            scope: openid,email
        provider:
          govuk-one-login:
            issuer-uri: https://oidc.integration.account.gov.uk/
```

**Session config (Express, OWASP-aligned):**

```js
app.use(session({
  cookie: { secure: true, httpOnly: true, sameSite: 'strict', maxAge: 8 * 60 * 60 * 1000 },
  rolling: true,          // resets idle timeout on activity
  resave: false,
  saveUninitialized: false,
}));
```

**GOV.UK One Login endpoints:**

| Environment | Discovery URL |
|-------------|--------------|
| Integration | `https://oidc.integration.account.gov.uk/.well-known/openid-configuration` |
| Production  | `https://oidc.account.gov.uk/.well-known/openid-configuration` |
