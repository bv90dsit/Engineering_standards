# ENG-004: Documentation for onboarding (README, ADRs, runbooks)

## Standard

Every repository SHOULD have a README that enables a new team member to understand the service, set up a development environment, and make their first contribution within one day. Production services SHOULD have operational runbooks.

## Rationale

- Government teams have high turnover (contractors, rotations, machinery of government changes)
- Poor documentation = weeks of lost productivity per new joiner

## What good looks like

- README with: what the service does, how to run locally, how to run tests, how to deploy, architecture overview, links to ADRs and runbooks
- ADRs stored in `docs/adr/` using the format `NNNN-title-with-dashes.md` (status: Proposed/Accepted/Deprecated/Superseded)
- Runbooks stored in `docs/runbooks/` with: per-alert response procedures, common tasks, escalation paths

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Automated (CI/CD)** | README.md exists in repo root | PR check |
| **Peer review** | Reviewer checks documentation updated when behaviour changes | Every PR |

**Primary enforcement: Automated + Peer review.** README.md exists in repo root; reviewer checks documentation updated when behaviour changes.

## Quick reference

**README template** (`README.md`):

```markdown
# Service Name

One-line description of what this service does.

## Architecture

Brief description or link to architecture diagram.
See [ADRs](docs/adr/) for decision records.

## Prerequisites

- Node.js 20+
- Docker (for local dependencies)
- Access to [vault/secrets location] for env vars

## Getting started

\```bash
git clone <repo-url>
cd <repo>
cp .env.example .env          # then fill in secrets from vault
npm install
npm run dev                    # starts on http://localhost:3000
\```

## Running tests

\```bash
npm test                       # unit tests
npm run test:integration       # integration tests (requires Docker)
\```

## Deployment

Deployed via [CI pipeline link]. See `docs/runbooks/deploy.md` for manual steps.

## Runbooks

- [Incident response](docs/runbooks/incident-response.md)
- [Common alerts](docs/runbooks/alerts.md)

## Contributing

See [CONTRIBUTING.md](CONTRIBUTING.md) for PR process and coding standards.
```

**ADR template** (`docs/adr/0001-use-postgres.md`):

```markdown
# 1. Use PostgreSQL for primary datastore

Date: 2026-01-15

## Status

Accepted

## Context

We need a relational database for [reason]. Options considered: PostgreSQL, MySQL, DynamoDB.

## Decision

Use PostgreSQL via AWS RDS.

## Consequences

- Good: JSONB support, strong ecosystem, team familiarity
- Bad: operational overhead vs managed NoSQL
```

**Create a new ADR quickly:**

```bash
# Install adr-tools (optional helper)
brew install adr-tools
adr new "Use PostgreSQL for primary datastore"
```
