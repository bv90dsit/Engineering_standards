# ENG-004: Documentation for onboarding (README, ADRs, runbooks)

## Standard

Every repository SHOULD have a README that enables a new team member to understand the service, set up a development environment, and make their first contribution within one day. Production services SHOULD have operational runbooks.

## Rationale

- Government teams have high turnover (contractors, rotations, machinery of government changes)
- Poor documentation = weeks of lost productivity per new joiner

## What good looks like

- README with: what the service does, how to run locally, how to run tests, how to deploy, architecture overview, links to ADRs and runbooks
- Runbooks: per-alert response procedures, common tasks, escalation paths

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Automated (CI/CD)** | README.md exists in repo root | PR check |
| **Peer review** | Reviewer checks documentation updated when behaviour changes | Every PR |

**Primary enforcement: Automated + Peer review.** README.md exists in repo root; reviewer checks documentation updated when behaviour changes.
