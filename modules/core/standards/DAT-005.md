# DAT-005: Audit trail for data changes

## Standard

Systems that store personal or sensitive data MUST maintain an audit trail recording who changed what, when, and why. Audit logs MUST be immutable and retained per the service's retention policy.

## Rationale

- UK GDPR Article 5(2) requires controllers to demonstrate compliance (accountability principle) — audit trails provide this evidence
- ICO investigations and subject access requests require the ability to show what data was held and when it changed
- Audit trails deter unauthorised changes and support forensic investigation after security incidents
- Immutability ensures audit records cannot be tampered with to cover malicious activity
- Regulatory requirements (GDPR, FOI, judicial review) may require proving the state of data at a specific point in time

## What good looks like

- Every create, update, and delete operation on personal/sensitive data recorded in an audit log
- Audit entries include: who (user ID or system actor), what (entity, field changed, old value, new value), when (timestamp), why (reason code or reference)
- Audit logs stored separately from application data — ideally in a write-only/append-only store
- Audit logs cannot be modified or deleted by application users or developers (immutability)
- Retention period defined and aligned with the service's data retention policy and legal requirements
- Audit logs queryable — can answer "show me all changes to this user's record" or "who accessed this data on this date"
- PII in audit logs subject to the same retention/deletion rules as source data (right to erasure considered)
- Audit mechanism tested — integration tests verify entries are created for all auditable operations

## What bad looks like

- No audit trail — no record of who changed personal data or when
- Audit logs stored in the same database table with application-level delete access
- Audit entries missing "who" (no user attribution) or "why" (no reason recorded)
- Audit logs that grow indefinitely with no retention policy
- Audit mechanism easily bypassed (e.g. direct database access not audited)
- Logging only successful changes — failed attempts and access events not recorded

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Peer review** | Audit logging present for all data mutation operations; audit entries include required fields (who, what, when, why) | Every PR modifying data models or persistence layer |
| **Periodic audit** | Audit log immutability verified; retention policy compliance checked; sample queries tested | Quarterly data governance review and annual DPIA review |

**Primary enforcement: Peer review + Periodic audit.** Code review ensures audit logging is implemented for data changes; quarterly reviews verify logs are immutable, queryable, and retained correctly.

## Quick reference

**Django audit log (django-auditlog):**

```python
# settings.py
INSTALLED_APPS = [
    ...
    'auditlog',
]

MIDDLEWARE = [
    ...
    'auditlog.middleware.AuditlogMiddleware',  # Captures acting user
]

# models.py
from auditlog.registry import auditlog
from django.db import models

class Application(models.Model):
    applicant_name = models.CharField(max_length=255)
    email = models.EmailField()
    status = models.CharField(max_length=50)
    updated_at = models.DateTimeField(auto_now=True)

# Register model for automatic audit logging
auditlog.register(Application)

# Every create/update/delete is now logged with:
# - actor (user from request middleware)
# - timestamp
# - changed fields (old value → new value)
# - action type (create/update/delete)
```

**PostgreSQL trigger-based audit:**

```sql
-- Audit table (append-only, no UPDATE/DELETE grants)
CREATE TABLE audit_log (
    id              BIGSERIAL PRIMARY KEY,
    table_name      TEXT NOT NULL,
    record_id       TEXT NOT NULL,
    action          TEXT NOT NULL CHECK (action IN ('INSERT', 'UPDATE', 'DELETE')),
    changed_by      TEXT NOT NULL,
    changed_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    old_values      JSONB,
    new_values      JSONB,
    reason          TEXT
);

-- Revoke modification rights (immutability)
REVOKE UPDATE, DELETE ON audit_log FROM app_user;

-- Audit trigger function
CREATE OR REPLACE FUNCTION audit_trigger_func()
RETURNS TRIGGER AS $$
BEGIN
    INSERT INTO audit_log (table_name, record_id, action, changed_by, old_values, new_values)
    VALUES (
        TG_TABLE_NAME,
        COALESCE(NEW.id::text, OLD.id::text),
        TG_OP,
        current_setting('app.current_user', true),
        CASE WHEN TG_OP IN ('UPDATE', 'DELETE') THEN to_jsonb(OLD) END,
        CASE WHEN TG_OP IN ('INSERT', 'UPDATE') THEN to_jsonb(NEW) END
    );
    RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql;

-- Apply to tables with personal data
CREATE TRIGGER audit_applications
    AFTER INSERT OR UPDATE OR DELETE ON applications
    FOR EACH ROW EXECUTE FUNCTION audit_trigger_func();
```

**Event sourcing pattern (conceptual):**

```
# Instead of storing current state, store the sequence of events:
#
# Event Store:
#   { event_id: 1, aggregate_id: "app-123", type: "ApplicationSubmitted", 
#     data: {name: "Jane Smith", email: "jane@example.com"}, 
#     actor: "user-456", timestamp: "2026-07-05T10:00:00Z" }
#
#   { event_id: 2, aggregate_id: "app-123", type: "ApplicationApproved",
#     data: {decision: "approved", reason: "meets criteria"},
#     actor: "assessor-789", timestamp: "2026-07-05T14:30:00Z" }
#
# The event store IS the audit trail — every change is an immutable event.
# Current state is derived by replaying events.
```
