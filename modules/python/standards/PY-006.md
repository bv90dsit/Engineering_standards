# PY-006: Flask secret key must not be hardcoded

## Standard

Flask `app.secret_key` and Django `SECRET_KEY` MUST be sourced from environment variables or a secrets manager, never hardcoded as string literals in source code.

## Rationale

- Hardcoded secrets are exposed to anyone with repository access (including CI logs, forks, and cached artifacts)
- Secret keys are used for session signing, CSRF tokens, and password reset links -- compromise enables session hijacking
- Secrets committed to version control persist in git history even after removal
- Environment-based secrets enable key rotation without code changes or deployments

## What good looks like

```python
# Flask
import os
app.secret_key = os.environ["FLASK_SECRET_KEY"]

# Django
SECRET_KEY = os.environ.get("DJANGO_SECRET_KEY")

# Or via a secrets manager
from myapp.secrets import get_secret
SECRET_KEY = get_secret("app/secret_key")
```

## What bad looks like

```python
# BAD: Hardcoded secret key
app.secret_key = "super-secret-key-do-not-share"

# BAD: Hardcoded in settings
SECRET_KEY = "django-insecure-abc123def456"
```

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Automated (CI/CD)** | Grep for `secret_key = "` or `SECRET_KEY = "` patterns in `.py` files | Every PR |

**Primary enforcement: Automated.** A regex-based scanner detects string literal assignments to secret_key or SECRET_KEY. Test files and example configuration files are excluded from this check.
