# PY-004: No wildcard imports

## Standard

Python code SHOULD NOT use `from module import *`. Explicit imports make dependencies clear and prevent namespace pollution.

## Rationale

- Wildcard imports make it impossible to determine where a name was defined
- They break IDE navigation (go-to-definition, find-references)
- Names can be silently shadowed when multiple wildcard imports are used
- Explicit imports create a clear dependency graph for the module

## What good looks like

- All imports are explicit: `from module import ClassA, function_b`
- Related imports are grouped (stdlib, third-party, local) per PEP 8
- `__all__` is defined in modules that are designed to be imported from
- Import statements are sorted (e.g. via `isort`)

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Automated (CI/CD)** | Regex for `from .+ import \*` in `.py` files | Every PR |

**Primary enforcement: Automated.** A linting rule (e.g. Ruff F403, Flake8 F403) or pre-commit hook flags any wildcard import. The check runs on all Python files in the repository.
