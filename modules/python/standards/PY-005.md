# PY-005: Django CSRF protection must not be disabled

## Standard

Django CSRF protection MUST NOT be disabled. The `@csrf_exempt` decorator, removing `CsrfViewMiddleware` from MIDDLEWARE, or setting `CSRF_COOKIE_SECURE = False` are all prohibited in production code.

## Rationale

- Cross-Site Request Forgery allows attackers to trick authenticated users into performing unintended actions
- Django's built-in CSRF middleware provides robust protection with minimal developer effort
- Every POST endpoint must include CSRF validation to prevent forged form submissions
- Disabling CSRF protection exposes the application to state-changing attacks from malicious sites

## What good looks like

- All POST/PUT/PATCH/DELETE endpoints are protected by Django's CsrfViewMiddleware
- `django.middleware.csrf.CsrfViewMiddleware` is present and active in MIDDLEWARE
- CSRF tokens are included in all forms via `{% csrf_token %}` template tag
- API endpoints that legitimately cannot use CSRF (e.g. token-authenticated APIs) use alternative authentication (DRF TokenAuthentication, JWT) rather than blanket `@csrf_exempt`

## What bad looks like

```python
# BAD: Disabling CSRF on a view
@csrf_exempt
def update_profile(request):
    ...

# BAD: Removing CSRF middleware
MIDDLEWARE = [
    # 'django.middleware.csrf.CsrfViewMiddleware',  # commented out
    ...
]

# BAD: Weakening CSRF cookie security
CSRF_COOKIE_SECURE = False
```

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Automated (CI/CD)** | Grep for `@csrf_exempt`, commented-out CsrfViewMiddleware, `CSRF_COOKIE_SECURE = False` | Every PR |

**Primary enforcement: Automated.** A linting rule or CI check scans Python files for patterns that disable or weaken CSRF protection. Exceptions require security team approval documented in the PR.
