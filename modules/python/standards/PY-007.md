# PY-007: Use Django ORM or SQLAlchemy, not raw SQL

## Standard

Application code SHOULD use Django's ORM querysets or SQLAlchemy's expression language instead of raw SQL via `cursor.execute()` with string formatting. This standard extends PY-003 with framework-specific guidance.

## Rationale

- Raw SQL with string formatting (`cursor.execute(f"SELECT ... WHERE id = {user_id}")`) is the primary vector for SQL injection
- Django ORM and SQLAlchemy's expression language automatically parameterise queries
- ORM-based queries are portable across database backends
- Framework query builders provide query composition, lazy evaluation, and built-in escaping
- Raw SQL bypasses model-level validation, signals, and audit hooks

## What good looks like

```python
# Django ORM
users = User.objects.filter(email=user_email, is_active=True)

# SQLAlchemy expression language
stmt = select(User).where(User.email == user_email)
result = session.execute(stmt)

# If raw SQL is absolutely necessary, use parameterised queries
from django.db import connection
with connection.cursor() as cursor:
    cursor.execute("SELECT * FROM users WHERE id = %s", [user_id])
```

## What bad looks like

```python
# BAD: String formatting in SQL
cursor.execute(f"SELECT * FROM users WHERE email = '{email}'")

# BAD: String concatenation
query = "SELECT * FROM orders WHERE user_id = " + str(user_id)
cursor.execute(query)

# BAD: .format() with SQL
cursor.execute("DELETE FROM sessions WHERE token = '{}'".format(token))
```

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Peer review** | Reviewers verify ORM usage and flag raw SQL patterns | Every PR |
| **Automated (SAST)** | Static analysis tools (Bandit B608, Semgrep) detect raw SQL construction | Every PR |

**Primary enforcement: Peer review + Automated.** SAST tools flag raw SQL string construction, and peer reviewers verify that any raw SQL uses parameterised queries with a documented justification.
