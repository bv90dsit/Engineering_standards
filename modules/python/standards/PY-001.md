# PY-001: Use logging module not print()

## Standard

Production Python code SHOULD use the `logging` module instead of `print()`. Print statements are acceptable in CLI scripts, management commands, and tests.

## Rationale

- The `logging` module provides severity levels (DEBUG, INFO, WARNING, ERROR, CRITICAL) enabling filtering by importance
- Logs can be routed to multiple handlers (files, syslog, monitoring systems) simultaneously
- `print()` goes to stdout only and cannot be filtered, formatted, or routed in production
- Structured logging enables correlation, searching, and alerting in observability platforms

## What good looks like

- A module-level logger: `logger = logging.getLogger(__name__)`
- Appropriate log levels used (INFO for business events, WARNING for recoverable issues, ERROR for failures)
- Logging configuration centralised at application entry point
- No `print()` statements in library or service code

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Automated (CI/CD)** | Regex check for `print(` in `.py` files excluding tests, CLI scripts, and management commands | Every PR |

**Primary enforcement: Automated.** A linting rule or pre-commit hook scans for `print(` in production code paths. Exceptions are configured for test files, CLI entry points, and management commands.
