# PY-003: No raw SQL string construction

## Standard

Python code MUST NOT construct SQL queries via string concatenation or f-strings. Use parameterised queries (e.g. `cursor.execute("SELECT * FROM x WHERE id = %s", (id,))`) or an ORM.

## Rationale

- SQL injection is OWASP A03:2021 — one of the most critical and prevalent web vulnerabilities
- String-built queries are the number one cause of SQL injection
- Parameterised queries separate code from data, making injection structurally impossible
- ORMs provide an additional abstraction layer that handles escaping automatically

## What good looks like

- All database queries use parameterised placeholders (`%s`, `?`, or named parameters)
- ORM usage (Django ORM, SQLAlchemy) for standard CRUD operations
- Raw SQL, when necessary, uses the ORM's raw query API with parameter binding
- No f-strings, `.format()`, or `+` concatenation in any SQL string

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Automated (CI/CD)** | SAST/regex detects patterns like `f"SELECT`, `"SELECT" +`, `"INSERT" +`, `.format(` adjacent to SQL keywords in `.py` files | Every PR |
| **Peer review** | Reviewer verifies any raw SQL uses parameterised queries | Every PR |

**Primary enforcement: Automated.** Static analysis (e.g. Bandit, Semgrep) flags string interpolation in SQL statements. Peer review provides a secondary check for edge cases that automated tools miss.
