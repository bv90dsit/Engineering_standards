# PY-002: Type hints on public function signatures

## Standard

Public functions and methods SHOULD have type annotations on parameters and return values. Private/internal functions (prefixed with `_`) are exempt.

## Rationale

- Type hints enable IDE autocompletion and inline documentation
- Static analysis tools (mypy, pyright) catch type-related bugs before runtime
- Annotations serve as machine-readable documentation of the API contract
- Reduces onboarding time for new team members reading unfamiliar code

## What good looks like

- All public functions have parameter and return type annotations
- Complex types use `typing` module constructs (`Optional`, `Union`, `list[str]`, etc.)
- `py.typed` marker file present in distributed packages
- mypy or pyright runs in CI with at least basic strictness

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Peer review** | Reviewer checks that public API functions have type hints | Every PR |
| **Automated (CI/CD, optional)** | `mypy --strict` or `pyright` in CI for annotated modules | Every PR |

**Primary enforcement: Peer review.** Reviewers verify that new or modified public functions include type annotations. Optionally, mypy with `--strict` mode in CI provides automated enforcement for modules that have opted in.
