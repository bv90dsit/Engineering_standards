# ARC-002: Twelve-factor application design

## Standard

Services SHOULD follow twelve-factor application principles. At minimum: configuration via environment variables, stateless processes, explicit dependency declaration, and port binding.

## Rationale

- Makes applications portable between environments (dev, staging, prod, cloud, on-prem)
- Enables horizontal scaling without application changes
- Simplifies deployment automation and container orchestration
- Reduces "works in dev, breaks in prod" failures

## What good looks like

- Configuration injected via environment variables or a secrets manager — never hardcoded
- Application is stateless; session state lives in external stores (Redis, database)
- Dependencies explicitly declared (requirements.txt, package.json, go.mod)
- Application binds to a port and serves HTTP — no reliance on app server configuration
- Logs written to stdout/stderr, collected by the platform
- Dev/prod parity maintained (same Docker image, different config)

## What bad looks like

- Secrets in source code or committed `.env` files
- Application state stored in local filesystem or in-memory between requests
- "It works if you install these 47 things on the server first"
- Logging to local files that no one reads

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Peer review** | No secrets hardcoded; config via env vars; stateless design | Every PR (reviewer checks) |
| **Periodic audit** | Full 12-factor assessment against production architecture | Service assessment |

**Primary enforcement: Peer review + Periodic audit.** Reviewers catch config-in-code and stateful anti-patterns during PR review. Broader architectural compliance (port binding, dev/prod parity, log routing) is assessed holistically during service assessment.
