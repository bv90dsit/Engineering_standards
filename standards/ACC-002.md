# ACC-002: Progressive enhancement

## Standard

Services SHOULD work without JavaScript enabled. Core functionality MUST be available as server-rendered HTML; JavaScript enhances but is not required for basic task completion.

## Rationale

- Users on low-bandwidth connections may not download JS bundles
- Older devices may not execute modern JavaScript
- Corporate proxies may strip or block JavaScript
- Some assistive technologies do not execute JavaScript well
- Improves resilience — service degrades gracefully rather than failing completely

## What good looks like

- Core journey works with JavaScript disabled
- Forms submit via standard HTML form actions
- JavaScript adds polish and enhanced interactions, not core functionality
- No blank white pages when JavaScript fails to load or execute
- Server-side rendering for initial page load

## What bad looks like

- Blank page with "Please enable JavaScript" message
- Single-page application with no server-side fallback for critical journeys
- Form validation only in JavaScript with no server-side validation
- Navigation that relies entirely on JavaScript event handlers

## Enforcement

| Mechanism | What is checked | When |
|-----------|----------------|------|
| **Peer review** | Reviewer tests core journey with JavaScript disabled | Every PR affecting user-facing pages |

**Primary enforcement: Peer review.** Reviewers verify that the core user journey functions with JavaScript disabled during code review. Automated testing of no-JS journeys may supplement this but is not the primary mechanism.
